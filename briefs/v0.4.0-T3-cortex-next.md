# Task: Implement `cortex next` deterministic MVP

You are an autonomous engineering agent dispatched by Cortex to implement
Tier 2 (v0.4.0) work item T3 from `.cortex/plans/cortex-v1.md`. You operate
inside a fresh git worktree branched off `origin/main`. You will own the
implementation end-to-end and ship it via `scripts/open-pr.sh --auto-merge`.

**This task depends on T1 (`cortex refresh-state`) and T2 (`cortex plan status`)
having merged to main.** Verify both have shipped before starting; if either is
still outstanding, abort and report.

## Goal

Implement `cortex next` — a deterministic command that produces a ranked list
of work items the user could pick up right now. Pulls from primary sources;
no LLM. This is case-study item #4 from
`docs/case-studies/2026-04-24-stale-claude-md-steered-agent-wrong.md` —
"automate the very compilation we did by hand."

## Required behavior

Walk three source classes and emit a ranked list:

**P0 — Active work (must-do candidates):**
- All open `[ ]` checkboxes from active non-stale plans (use `cortex plan
  status` parser; exclude stale plans from P0 — they go to P1).
- Items in `.cortex/state.md` `## Current work` section that contain a
  checkbox or bullet (parse the section body; use the marker-aware reader
  from T1 if `<!-- cortex:hand -->` wraps the section).

**P1 — Open questions and stale debt:**
- Items in `.cortex/state.md` `## Open questions` section.
- Open `[ ]` checkboxes from stale active plans (15+ days since last
  Updated-by) — these need attention but the staleness flag implies the work
  may need re-scoping first.

**P2 — Recent context to consider:**
- `docs/case-studies/*.md` modified in the last N days (default 30,
  configurable via `--since <DAYS>`). One bullet per case study, with the
  one-line summary from the file's first paragraph after the H1.

Each item carries a stable citation: the file path and either a line range
(for checkbox items) or a section header anchor (for state.md items).

## Output format

Default human output, grouped by priority band:

```
P0 — Active work
  • plans/cortex-v1.md:67   Deterministic cortex refresh-state
  • plans/cortex-v1.md:68   cortex next deterministic MVP
  • state.md#current-work   Three-target dogfood gate

P1 — Open questions and stale debt
  • state.md#open-questions Marker convention for refresh-state

P2 — Recent context to consider
  • docs/case-studies/2026-04-24-stale-claude-md-steered-agent-wrong.md
     stale CLAUDE.md tap claim steered an agent to recommend the wrong
     install path
```

`--json` emits an object with three keys (`p0`, `p1`, `p2`), each a list of
items with `text`, `source`, `line_start`, `line_end` (when applicable).

`--limit N` truncates each band to at most N items (default: no limit).

`--path <DIR>` targets an arbitrary `.cortex/`-bearing project.

## Implementation outline

- New module: `src/cortex/commands/next.py` — Click command.
- Reuses parsers from T1 (state.md walker) and T2 (`cortex plan status`).
- Pure ranking logic in `src/cortex/ranking.py` or inlined; choose the
  shape that keeps the test surface small.
- Stable ordering: within each priority band, items sort by source path
  then line number. Output must be byte-identical across two runs on
  unchanged inputs (tested).

## Tests required

Place tests under `tests/test_next.py`. Use `tmp_path` fixtures with a
synthetic `.cortex/` and `docs/case-studies/`.

Required test cases:

1. **Stable ordering.** Two consecutive runs produce byte-identical output.
2. **P0 contents.** Active plans contribute open checkboxes; shipped plans
   do not; stale plans go to P1, not P0.
3. **P1 contents.** Open questions section parsed; stale-plan open checkboxes
   appear here, not P0.
4. **P2 contents.** Case studies older than `--since` excluded; newer
   included; one bullet per file with the summary line.
5. **Citation correctness.** Cited line numbers match source files.
6. **Empty case.** Project with no active plans, no open questions, no
   recent case studies — output is the three section headers with "(none)"
   under each.
7. **`--json`** is parseable and structurally matches the documented shape.
8. **`--limit`** truncates correctly.
9. **`--path`** targets an arbitrary project.

## Acceptance criteria

- `cortex next` runs from this repo's root and produces a non-empty P0
  band on the current state of `.cortex/`.
- All existing tests pass; new tests pass under
  `bash scripts/touchstone-run.sh validate`.
- `cortex doctor` continues to pass.
- The PR description shows the actual output of `cortex next` on this repo
  as a sanity check the dispatcher (the user) can eyeball.
