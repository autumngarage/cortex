# Task: Touchstone post-merge hook → cortex journal draft pr-merged

You are an autonomous engineering agent dispatched to implement Tier 3
(v0.5.0) work item T2 from `.cortex/plans/cortex-v1.md`. This is a
**cross-repo task**: the hook lives in `autumngarage/touchstone`; the
Cortex side is a small verification that `cortex journal draft pr-merged
--no-edit` works end-to-end.

You operate inside a fresh git worktree branched off `origin/main` of
**touchstone** (not cortex). The dispatch wrapper has cloned touchstone
into your worktree path; do not assume the cortex repo layout.

## Goal

Wire a Touchstone post-merge hook that, on default-branch merges,
shells out to `cortex journal draft pr-merged --no-edit` so every merge
becomes a journal entry without user action. This is the inflection-point
automation that turns Cortex from "templated journaling tool" into
"install-and-walk-away repo memory."

## Required behavior

1. **New hook entry** in `.pre-commit-config.yaml` (or wherever Touchstone's
   hook registry lives — grep for `merge-pr.sh` to locate the relevant
   stage). Hook fires at `post-merge` stage on the default branch only.

2. **Opt-in per project** via a `.touchstone-config` field:

   ```toml
   [cortex]
   journal_on_merge = true   # default false
   ```

   When `false` or absent, hook is a no-op.

3. **Graceful degradation.** If `cortex` is not on PATH:
   - Print one stderr line: `[touchstone] cortex CLI not on PATH; skipping
     pr-merged journal entry`
   - Exit 0 (do not block the merge).

4. **When cortex is present and config opts in:**
   - Run `cortex journal draft pr-merged --no-edit` in the merged repo's
     working directory.
   - Capture stdout (the journal entry path) and print one line:
     `[touchstone] wrote .cortex/journal/<filename>.md`
   - Exit 0 regardless of cortex's exit code (the hook is observational;
     a failure to journal must not block a merge).

5. **Idempotency.** Multiple invocations on the same merge produce one
   entry per invocation (each entry timestamped uniquely). This is
   acceptable; the post-merge hook fires once per merge by design.

## Implementation outline

- Locate Touchstone's hook framework. The typical pattern is a shell
  script under `scripts/` invoked from `.pre-commit-config.yaml` or from
  the merge driver. Grep for `post-merge` and `merge-pr` to find the
  surface.
- New script: `scripts/cortex-journal-merge.sh` (or wherever fits the
  existing pattern). Bash; reads `.touchstone-config` via `tomlq` if
  present (Touchstone already has TOML parsing somewhere — find it),
  shells out to `cortex journal draft pr-merged --no-edit`.
- Register in `.pre-commit-config.yaml` with `stages: [post-merge]` and
  the appropriate branch filter (default branch only).
- Update Touchstone's README / hook docs to mention the new opt-in.

## Tests required

Touchstone's test convention should be followed (grep for existing test
files). The hook must be tested with:

1. **Cortex absent.** `PATH` strip; hook exits 0 with the stderr message.
2. **Cortex present, config off.** Default `.touchstone-config` (no
   `[cortex]`); hook is a no-op.
3. **Cortex present, config on.** Synthesize a merge in a temp git repo
   that has a `.cortex/` scaffold; assert a `pr-merged` journal entry
   appears.

## Acceptance criteria

- Hook fires on the next merge to touchstone's main when the opt-in is
  set in `.touchstone-config`.
- All existing Touchstone tests pass.
- README / hook docs updated.
