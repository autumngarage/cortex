## Required behavior — eight new checks

For each check below, add a function in `src/cortex/doctor_checks.py`
(creating that module if needed; consolidate existing doctor logic into
it). Each check takes a `Path` and returns a list of `Finding` instances.

### 1. Append-only Journal violation (SPEC § 3.5)

Walk `git log --follow .cortex/journal/<file>.md` for every journal file.
If any commit modifies an existing journal file (vs. adding it), emit a
warning: `journal <file> modified at commit <sha> — append-only invariant
violated`. The check is best-effort: a file moved (rename) does not count
as a modification. Exception: changes to YAML frontmatter `Updated-by:`
are permitted by SPEC § 3.5; detect frontmatter-only diffs and skip.

### 2. Immutable-Doctrine mutation (SPEC § 3.1)

Same shape as #1, but for `.cortex/doctrine/*.md`. Doctrine entries are
immutable except via Superseded-by. A non-superseded entry that's been
modified post-creation is a violation. Allow `Status:` field changes
(`Active` → `Superseded-by Nxxx`) and frontmatter-only diffs.

### 3. Promotion-queue invariants (SPEC § 4.7)

Read `.cortex/.index.json` via T1's helper (`cortex.index_json`). Verify:
- Every `candidates[].source` resolves to a real file under `.cortex/journal/`.
- Every `candidates[].promoted_to` (when set) resolves to a real file under
  `.cortex/doctrine/`.
- No two candidates share the same `id`.
- `age_days` matches `today - last_touched` within 1 day.

### 4. CLI-less-fallback warning (Protocol § 1)

If the project lacks a `.cortex/.index.json` AND has more than 20 Doctrine
entries OR more than 100 Journal entries, warn that the corpus exceeds the
fallback threshold and `cortex refresh-index` should run.

### 5. T1.4 file-deletion audit (Protocol § 2)

Walk `git log` for the audit window (default 7 days). For every commit that
deleted a file with more than the configured threshold (default 100 lines)
that isn't matched by a `Type: decision` Journal entry within 72 h of the
commit, emit a warning. The threshold is configurable via
`.cortex/protocol.md` per-project override (already documented in
Protocol § 2). Reuse the existing audit-window logic in
`src/cortex/audit.py`.

### 6. Generated-layer contract validation (SPEC § 4.3) — NEW (codesight + council add)

Every file regenerated by `cortex refresh-state` or `cortex refresh-index`
MUST carry frontmatter with `Generated:`, `Generator:`, `Sources:`,
`Corpus:`, `Omitted:`, `Incomplete:`, `Conflicts-preserved:` (per SPEC § 4.3).
Walk all files matching the regenerated-layer pattern (currently
`.cortex/state.md`, `.cortex/map.md` if present, and any `.cortex/digests/`
files). For each:
- If frontmatter is missing entirely → error.
- If any required field is absent → warning citing the missing field.
- If `Generated:` is older than `cortex refresh-state`'s expected freshness
  (configurable, default 7 days) → warning ("layer is stale; rerun
  `cortex refresh-state`").

### 7. `.cortex/config.toml` schema validation — NEW (codesight + council add)

If `.cortex/config.toml` exists, parse it (use `tomllib` from stdlib, py3.11+).
Validate against the known schema for `[audit-instructions]` and
`[doctrine.0007]` sections:
- `[audit-instructions] homebrew_tap` — string or null.
- `[audit-instructions] siblings` — list of strings or null.
- `[audit-instructions] pypi_package` — string or null.
- `[audit-instructions] gh_release` — string or null.
- `[audit-instructions] urls` — list of strings or null.
- `[doctrine.0007] allowed_root_files` — list of strings (default `[]`).

Unknown keys at the section level → warning naming the key. Type mismatches
→ error with line number from the toml parse exception when available.
Catches misconfigured blocks before `--audit-instructions` runs and silently
breaks.

### 8. SPEC § 5.1 retention visibility — NEW (codesight + council add)

Visibility, not destructive cleanup. Walk:
- `.cortex/plans/*.md` with `Status: shipped` or `Status: cancelled`. If
  `Date:` (or first commit timestamp if no Date:) is older than the
  default 90-day plan-archive threshold, warn ("plan eligible for archive
  to `.cortex/plans/archive/`; v0.6.0 surfaces this; destructive cleanup
  is parked v1.x").
- `.cortex/journal/*.md` files older than the warm-tier threshold (default
  90 days). If the count exceeds the configurable `journal_warm_max`
  (default 200), warn ("warm journal threshold exceeded; consolidation to
  digest is overdue per SPEC § 5.1").

This is a warning-only check. No file moves, no deletions. The point is
visibility into retention pressure; the destructive-cleanup automation is
deferred per Doctrine 0005 conservatism + the council de-scope notes.

### 9. Canonical-ownership warning — NEW per Doctrine 0007 (recursive-failure add 2026-05-02)

This check is the production form of [Doctrine 0007](../.cortex/doctrine/0007-canonical-ownership-of-state-and-plans.md).
Read the full doctrine before implementing — the rule, bounds, and override
mechanism are all there.

**Trigger condition:** scan repo root for files matching the case-insensitive
regex `^(ROADMAP|STATUS|PLAN|PLANS|NEXT|TODO)\.md$` (so `ROADMAP.md`,
`roadmap.md`, `Roadmap.md` all match). Fire ONLY when:
- `.cortex/state.md` exists at the project's `.cortex/` root, AND
- At least one `.cortex/plans/*.md` file has `Status: active` in frontmatter.

**When triggered, emit a warning** naming each offending file:

```
WARNING canonical-ownership: ROADMAP.md at repo root duplicates
content canonical in .cortex/state.md and .cortex/plans/<active>.md.
Per Cortex Doctrine 0007, the canonical answers to "where are we"
and "what's next" live in .cortex/. README links here, never restates.
Either link from README and slim/delete this file, OR if you have a
documented reason to keep it, add to .cortex/config.toml:
  [doctrine.0007]
  allowed_root_files = ["ROADMAP.md"]
See: https://github.com/autumngarage/cortex/blob/main/.cortex/doctrine/0007-canonical-ownership-of-state-and-plans.md
```

**Override mechanism:** if `.cortex/config.toml` `[doctrine.0007]
allowed_root_files` includes the matched filename (case-sensitive against
the actual filename, not the regex group), suppress the warning for that
file. The override is per-file, explicit, never silent.

**Out-of-scope filenames** (do NOT warn on these — they have legitimate
canonical roles per Doctrine 0007 § Bounds):
- `CHANGELOG.md`, `RELEASES.md` — release notes are canonical at root.
- `README.md`, `ARCHITECTURE.md` — overview, architecture.
- `CONTRIBUTING.md`, `CODE_OF_CONDUCT.md`, `LICENSE.md`, `SECURITY.md` — governance.
- `AGENTS.md`, `CLAUDE.md`, `GEMINI.md` — agent-instruction files.
- Any subdirectory file (only repo root is scanned — `docs/roadmap.md` does
  not trigger).

This check runs on plain `cortex doctor` (always-on, like the orphan-deferral check).

## Implementation outline

- New module: `src/cortex/doctor_checks.py` consolidates check functions.
- Wire into `src/cortex/commands/doctor.py`. Each check is gated by either
  the existing `doctor` invocation or the existing `--audit` /
  `--audit-digests` / `--audit-instructions` flags as appropriate.
  - #1, #2, #4, #6, #9 run on plain `cortex doctor` (always).
  - #3, #5, #7, #8 run under `--audit`.
- All new checks emit warnings (exit 0); under `--strict`, warnings exit 1.
- For #9 (canonical-ownership), the regex constants and override-section name
  go in module-level constants so they're testable + visible.
