# Task: SPEC-to-test traceability matrix (v1.0 work item)

You are an autonomous engineering agent dispatched by Cortex to produce
the SPEC-to-test traceability matrix at `docs/spec-conformance.md`. Per
Doctrine 0003 (SPEC is the primary deliverable), this is the proof that
the CLI implements the spec — not the other way around.

You operate inside a fresh git worktree branched off `origin/main`. You
will own the work end-to-end and ship via `scripts/open-pr.sh --auto-merge`.

This is the sharpest single v1.0 council recommendation (per
`.cortex/journal/2026-04-28-codesight-cross-pollination-and-council-review.md`).
Any normative § with no check or test is a v1.0 risk to surface.

## Goal

Produce `docs/spec-conformance.md` mapping every normative § in
`SPEC.md` to either:
- a `cortex doctor` check (name the check + file it lives in),
- a test assertion in the cortex test suite (name the test + file),
- a documented "deferred to vN.x" note (with citation to the parking
  journal entry).

The output is a markdown table organized by SPEC § number. Every cell is
either a citation or an explicit "deferred" with reason.

## Method

1. Read `SPEC.md` end-to-end. Identify every normative requirement —
   look for "MUST", "MUST NOT", "REQUIRED", "fail `cortex doctor`",
   "violations surface as warnings or errors", and similar contractual
   language. The non-normative narrative (rationale, examples, footnotes)
   doesn't need a row.
2. For each requirement, grep the cortex source + tests:
   - `grep -rn "<keyword>" src/cortex/` — find the check function.
   - `grep -rn "<keyword>" tests/` — find the test.
   - If a section has multiple sub-requirements, give each its own row.
3. For requirements with no check + no test:
   - Check the deferred-items list in `.cortex/plans/cortex-v1.md`
     `## Follow-ups (deferred)` — many parked items document the gap.
   - Cite the parking journal entry by relative path.
   - If no parking citation exists for a real gap, flag it in the matrix
     as `**GAP — no parking entry**` so the v1.0 ceremony can either
     fix it or formally park it.

## Required structure for `docs/spec-conformance.md`

```markdown
# SPEC-to-test traceability matrix

> Per [Doctrine 0003](../.cortex/doctrine/0003-spec-is-the-artifact.md),
> SPEC is the primary deliverable. This matrix is the proof that the
> reference CLI implements the spec — every normative § maps to a test,
> a doctor check, or a documented deferral.
>
> Last updated: <YYYY-MM-DD>. Regenerated whenever SPEC.md changes;
> drift surfaces in `cortex doctor --audit-spec` (planned for v1.x).

## How to read this matrix

| SPEC § | Requirement | Enforcement | Test | Notes |
|---|---|---|---|---|
| § N.M | one-line restatement of the requirement | `check_<name>` in `src/cortex/<file>:<line>` OR n/a | `test_<name>` in `tests/<file>` OR n/a | citation if deferred |

A row with both Enforcement = n/a AND Test = n/a is a gap — must have a
"deferred to vN.x" note in Notes.

## Matrix

### § 2 Directory layout
| ... | ... | ... | ... | ... |

### § 3 Layer contracts
(one subsection per layer: Doctrine § 3.1, Map § 3.2, State § 3.3,
Plans § 3.4, Journal § 3.5, Procedures § 3.6)

### § 4 Cross-layer rules
(one subsection per § 4.N)

### § 5 Retention and consolidation
(per § 5.1, § 5.2, § 5.3, § 5.4)

### § 6 File format conventions

### § 7 Versioning

## Gaps surfaced

(if any rows above are flagged GAP, list them here as a single bullet
each, with proposed disposition: ship-fix-in-v1.0 or formally-defer-to-vN.x)
```

## Acceptance criteria

- Every normative § in SPEC.md has at least one row in the matrix.
- Every row has either Enforcement OR Test OR a deferred citation —
  never blank.
- All citations resolve (tests exist at the named line; doctor checks
  exist at the named file/function; deferred citations point to real
  journal entries / plan sections).
- A new file `docs/spec-conformance.md` is created. README.md gets a
  one-line link in its `## Documentation` or `## Status` section.
- `cortex doctor` doesn't surface new errors after this PR.
- No code changes. No test changes (unless the matrix itself reveals a
  trivial-to-add test that closes a gap — those should be separate
  follow-up PRs, not bundled here).

## Out of scope

- **Closing gaps with new code.** If the matrix reveals a missing
  enforcement or test, document it as a deferred item with a proposed
  disposition. Closing the gap is separate work — bundling it here would
  conflate "audit" with "fix" and bloat the PR.
- **`cortex doctor --audit-spec` automation.** That's the v1.x
  follow-up — the matrix today is hand-authored from a snapshot read of
  SPEC + code. Future automation regenerates it from a SPEC parse.

## SPEC + design references
