# Task: `.cortex/config.toml` schema reference doc (v1.0 work item)

You are an autonomous engineering agent dispatched by Cortex to write the
public-facing reference for the `.cortex/config.toml` schema. This is one
of three council-added v1.0 work items — a user-facing config format
without published schema is an adoption wall.

You operate inside a fresh git worktree branched off `origin/main`. You
will own the work end-to-end and ship via `scripts/open-pr.sh --auto-merge`.

## Goal

Produce `docs/config-reference.md` documenting every key under every
section of `.cortex/config.toml`, with expected types, defaults, and a
worked example. This is read code, write docs — no implementation work.

## Scope

The canonical source of truth for what config keys exist + their types is
`src/cortex/config.py` (the dataclasses + parsing). Read that file +
`src/cortex/doctor_checks.py` (the schema validation in
`check_config_toml_schema`) + the v0.6.0 release journal entry to find
the full key list.

Sections to document (verify against code; this is the expected list):

### `[audit-instructions]`
- `homebrew_tap` — string | null. Default: null. The Homebrew tap to
  audit for stale claims (e.g., `"autumngarage/cortex"`).
- `siblings` — list of strings | null. Default: empty. Sibling-repo
  references to cross-check.
- `pypi_package` — string | null. Default: null. PyPI package name.
- `gh_release` — string | null. Default: null. GitHub release URL pattern.
- `urls` — list of strings | null. Default: empty. Free-form URLs to check.
- `scan_files` — list of strings | null. Default: cortex's
  `DEFAULT_AUDIT_SCAN_FILES`. Files to scan for external claims (typically
  `["CLAUDE.md", "AGENTS.md", "README.md", "PITCH.md"]`).
- `github_repos` — list of strings | null. Default: empty. GitHub repos
  to check for release / tap state.

### `[doctrine.0007]`
- `allowed_root_files` — list of strings. Default: `[]`. Repo-root
  filenames that suppress the canonical-ownership warning per Doctrine
  0007 (e.g., `["ROADMAP.md"]` if a project has a documented reason to
  keep a root-level roadmap file). Per-file, case-sensitive against the
  actual filename.

### Other sections

Verify against `src/cortex/config.py` whether any other top-level
sections exist. If yes, document them with the same shape. If no,
explicitly state that the file is currently `[audit-instructions]` +
`[doctrine.0007]` only and that future sections will be additive (per
SPEC § 7 versioning rules).

## Required structure for `docs/config-reference.md`

```markdown
# `.cortex/config.toml` reference

> The optional per-project configuration file. All sections + keys are
> optional; an absent file uses defaults throughout.

## File location

`<project-root>/.cortex/config.toml`. Never required for `cortex doctor`,
`cortex manifest`, etc. to work — defaults cover the no-config case.

## Compatibility

Cortex SPEC v0.5.x + the v0.6.0+ CLI accept this format. Future sections
will be additive per SPEC § 7. Unknown keys at the section level surface
as warnings via `cortex doctor` (not errors); type mismatches surface as
errors.

## Sections

### `[audit-instructions]`

(per-key documentation as above; one row per key with type, default, and
1-2 sentence description; cite where in the cortex source the key is
parsed)

### `[doctrine.0007]`

(same shape)

## Worked example

A complete realistic config for a project that has a Homebrew tap and
wants to allow a root-level `ROADMAP.md`:

```toml
[audit-instructions]
homebrew_tap = "autumngarage/example"
siblings = ["autumngarage/example-helper"]
pypi_package = "example"
gh_release = "https://github.com/autumngarage/example/releases"
urls = ["https://example.com/install"]
scan_files = ["CLAUDE.md", "AGENTS.md", "README.md"]
github_repos = ["autumngarage/example"]

[doctrine.0007]
allowed_root_files = ["ROADMAP.md"]
```

## Validation

`cortex doctor` validates this file via `check_config_toml_schema` (see
`src/cortex/doctor_checks.py`). Run `cortex doctor` to surface unknown
keys (warning) or type mismatches (error).
```

## Acceptance criteria

- File exists at `docs/config-reference.md` with all sections + keys
  documented from the actual current code (not invented).
- Every key documented matches what `src/cortex/config.py` parses + what
  `src/cortex/doctor_checks.py` validates. Zero drift between docs and
  code.
- Worked example at the bottom is a valid, parseable config that
  passes `cortex doctor`.
- README.md gets a one-line link to the new doc in its `## Configuration`
  section (or the appropriate place — find or create).
- No code changes; no test changes. This is a docs PR.

## Out of scope

- New config keys. If you find a key in the code that arguably should
  exist but doesn't, file a follow-up issue; don't add it here.
- SPEC.md edits. The schema reference is a CLI-side doc, not part of
  SPEC.md.
