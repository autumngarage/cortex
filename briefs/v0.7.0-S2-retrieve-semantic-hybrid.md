# Task: `cortex retrieve --mode semantic` + default `--mode hybrid` (v0.7.0 S2)

You are an autonomous engineering agent dispatched by Cortex to implement
Slice S2 of the v0.7.0 retrieve interface from
`.cortex/plans/cortex-retrieve.md`. You operate inside a fresh git worktree
branched off `origin/main`. You will own the implementation end-to-end and
ship it via `scripts/open-pr.sh --auto-merge`.

S1 (BM25/FTS5 layer) shipped in v0.7.0 (PR #83). Your work builds on top of
the existing `src/cortex/retrieve/` package and `cortex retrieve` command.
This release ships as v0.8.0.

## Goal

Add semantic retrieval via sqlite-vec + fastembed + hybrid mode that
reciprocal-rank-fuses BM25 and semantic results. After this slice ships,
`cortex retrieve` defaults to hybrid mode (with a one-time notice on first
flip), and the cross-platform install-risk surface is documented + handled
by the bare-repo degradation fixture.

## Required behavior

### CLI surface

```
cortex retrieve <query> [--mode {hybrid,bm25,semantic}] [--top-k N] [--json] [--no-rebuild]
```

- `--mode hybrid` (NEW; becomes default in this slice). Reciprocal-rank
  fusion of BM25 and semantic results. RRF k=60 unless dogfood evidence
  shows otherwise; document the choice.
- `--mode semantic` (NEW). Pure cosine similarity over the embeddings table.
- `--mode bm25` (existing from S1). Unchanged.
- **Default mode flip:** when the chunks index has been built only at S1
  (no embeddings table yet), continue defaulting to `bm25` until the
  embeddings table is populated. After embeddings exist, default flips to
  `hybrid`. **Emit a one-line stderr notice on the first invocation that
  flips** so users see the behavior change. Don't silently change behavior.

### Index extension

Extend the existing `chunks.sqlite` schema with an `embeddings` virtual
table backed by sqlite-vec:

```sql
CREATE VIRTUAL TABLE embeddings USING vec0(
  embedding float[384]   -- BAAI/bge-small-en-v1.5 dimension
);
-- Per-row: rowid matches chunks.id; insertion driven by chunks insert/update.
-- Add embedding_model TEXT to meta table (e.g. 'bge-small-en-v1.5')
-- Add embedding_dim INTEGER to meta table
```

Bump the schema_version meta key. On opening an index built at the prior
version, if no `embeddings` table exists, build it incrementally — same
mtime+size invalidation logic as the chunks table, but per-chunk: an
embedding is recomputed only when the chunk content changed.

### Embedding model

- **Model:** `BAAI/bge-small-en-v1.5` via `fastembed` (CPU-only ONNX). Pin
  the version in `pyproject.toml`. Document the choice + the alternatives
  considered in a comment in `src/cortex/retrieve/embeddings.py`.
- **Dimension:** 384.
- **Cache path:** force fastembed cache to `~/.cache/cortex/models/`
  (XDG-respectful) — fastembed's default can collide with permissions on
  some systems (council delta #4). Project-level override:
  `.cortex/.index/models/` if `~/.cache` is unavailable. Detect via
  `os.path.expanduser` + writability check at first model load; fall back
  with a one-line stderr notice.
- **Tokenizer:** use fastembed's bundled tokenizer (matches the model).
  When this lands, replace the word-count×1.3 token estimate in
  `src/cortex/retrieve/chunker.py` with the real tokenizer count.

### Hybrid mode (RRF)

For a query Q, fetch top K results from BM25 and top K from semantic
(default K = top_k × 3 to give the fusion some room). Compute fused score:

```
score(d) = sum over rankers of [1 / (k + rank_in_ranker(d))]   where k=60
```

Return top top_k by fused score. Document the choice of k=60 in a
comment (industry default; Microsoft Azure AI Search uses it).

### Bare-repo / missing-runtime fallback (council requirement)

Per the brief and v0.9.0 dogfood gate: if sqlite-vec extension fails to
load OR fastembed/onnxruntime fails to import OR the embedding model fails
to download:

1. Detect at first-use of semantic/hybrid mode — try the import + a tiny
   end-to-end probe (embed "test" against the model).
2. Print a clear actionable stderr message, e.g.:
   `"sqlite-vec extension not available; falling back to BM25-only. Install
   sqlite-vec via 'pip install sqlite-vec' or use 'cortex retrieve --mode bm25'
   directly. Cross-platform note: aarch64 Linux lacks onnxruntime PyPI wheels."`
3. **Fall back to BM25** for this query and any subsequent semantic/hybrid
   query. Cache the failure-mode flag in-process to avoid re-probing.
4. Update `cortex doctor` to surface the runtime-availability check on
   plain `cortex doctor` (not just `--audit`), so users know up front
   whether their machine can do semantic retrieval. Warning, not error.
5. Bare-repo degradation fixture (already in v0.9.0 scope) extends to
   verify all four paths: sqlite-vec missing, fastembed missing, model
   download failed, model present + working.

This is the "no silent failures" principle. The user must know when
they're getting BM25 only.

### `--no-rebuild` semantics extend

When `--no-rebuild` is set, skip BOTH the chunks-index incremental rebuild
AND the embeddings backfill. Stale results acceptable; that's what the
flag asks for.

## Implementation outline

- New module: `src/cortex/retrieve/embeddings.py` — model-loading,
  embedding generation, fastembed cache management.
- Update `src/cortex/retrieve/index.py` — add `embeddings` virtual table,
  schema migration from S1, per-chunk embedding insertion on chunk
  insert/update.
- Update `src/cortex/retrieve/query.py` — add semantic + hybrid query
  paths. RRF fusion lives here. The BM25 path stays untouched.
- Update `src/cortex/commands/retrieve.py` — extend `--mode` enum to
  accept hybrid/semantic; flip default-mode logic; emit the one-time
  notice.
- Update `pyproject.toml` — add `sqlite-vec` and `fastembed` as deps.
  Pin versions. Document the cross-platform install footprint in the
  same commit.
- Update `src/cortex/commands/doctor.py` — add a runtime-availability
  check (warning) for semantic retrieval.
- Update `.cortex/.gitignore` — verify `.index/models/` is also covered
  if you use the per-project model cache.

## Tests required

`tests/test_retrieve_semantic.py` (new) and add to
`tests/test_retrieve_bm25.py` for the integration path.

1. **Schema migration from S1.** Build a chunks.sqlite at S1 schema (no
   embeddings table). Open with S2 code → embeddings table is created +
   backfilled. Existing BM25 results unchanged.
2. **Semantic query returns relevant hits.** Tiny corpus of 5 markdown
   files with overlapping topics; semantic query for "vector store doctrine"
   should rank the doctrine entry above the unrelated journal entry.
3. **Hybrid mode RRF math.** Synthetic ranker outputs (mock the BM25 +
   semantic returns); verify fused ordering matches the documented RRF
   formula with k=60.
4. **Default mode flip + one-time notice.** Build a fresh chunks index
   (no embeddings yet) → default is bm25, no notice. Backfill embeddings
   → next call's default is hybrid + stderr emits the notice once.
   Subsequent calls don't re-emit.
5. **Embedding cache path forced.** Mock fastembed's cache discovery;
   verify it's directed at `~/.cache/cortex/models/` (or per-project
   fallback if HOME unwritable) regardless of fastembed's default.
6. **Bare-repo fallback paths.** Mock each failure point separately:
   - sqlite-vec import fails → stderr notice + BM25 fallback for this
     query AND subsequent ones in same process.
   - fastembed import fails → same shape.
   - Model download fails → same shape.
   - All present + working → no notice, semantic/hybrid works.
7. **`cortex doctor` runtime warning.** When sqlite-vec or fastembed
   isn't importable, plain `cortex doctor` surfaces a warning naming the
   missing dep + actionable install hint.
8. **`--no-rebuild` skips embedding backfill.** Build chunks but force
   embeddings stale → with `--no-rebuild`, no backfill runs, stderr
   notice that results may be stale.
9. **`--json` schema unchanged.** The `[{path, score, frontmatter,
   excerpt}]` shape stays identical regardless of mode. Only the score
   meaning differs (BM25 score vs cosine vs RRF fused).
10. **Latency budget.** On the 200-entry synthetic corpus, hybrid query
    completes in <1.5s warm (CI-tolerant: <4s). Cold rebuild + first
    query <15s including model download (CI-tolerant: skip the cold
    test or use a cached model fixture).

  that's a separate slice.
