# Task: v0.5.1 polish — grep templates, --version flag, ecosystem-aware refresh-state

You are an autonomous engineering agent dispatched by Cortex to address
three small quality-of-life findings from the autumn-mail dogfood of
v0.5.0 (2026-04-28). You operate inside a fresh git worktree branched
off `origin/main`. You will own the fix end-to-end and ship via
`scripts/open-pr.sh --auto-merge`.

## Goal

Three independent but small fixes, bundled into one PR because each is
~20 lines and they share the "found-during-dogfood" theme.

## Required fixes

### 1. `cortex grep --frontmatter` excludes `templates/` by default

**Problem.** `cortex grep "" --frontmatter Type:decision` matches the
journal template (`.cortex/templates/journal/decision.md`) because the
template literally contains `Type: decision`. Templates are
scaffolding, not user content; including them produces false positives
on every frontmatter filter the templates happen to satisfy.

**Fix.** Skip `.cortex/templates/` by default in `cortex grep`. Add
`--include-templates` flag to opt back in for users who want to grep
the template library itself.

**Implementation.** In `src/cortex/commands/grep.py`, when computing
`search_root` and during the rg invocation, pass
`--glob '!.cortex/templates/**'` as default. Add `--include-templates`
boolean flag that disables the exclusion. The exclusion ALSO applies
to filter-only mode (no PATTERN, --frontmatter only).

**Tests.** Extend `tests/test_grep.py`:
- Default frontmatter filter run skips `.cortex/templates/`.
- `--include-templates` matches template files.
- A user who passes `--layer templates` explicitly searches only
  templates (the existing `--layer templates` overrides the
  exclusion). This is the deliberate-search path.

**Help + docs.** Update `cortex grep --help` and `docs/grep.md` (the
file shipped with PR #63). One-paragraph addition explaining the
default and the opt-in flag.

### 2. `cortex --version` flag (in addition to `cortex version`)

**Problem.** `cortex --version` errors with `No such option: --version`.
`cortex version` (subcommand) is the only path. Standard CLI
convention is to support both.

**Fix.** Add `@click.version_option(__version__, "-V", "--version")`
to the root `cli` group in `src/cortex/cli.py`. Output should match
the short form of the existing `cortex version` subcommand: just
`cortex 0.5.x` (one line). The richer `cortex version` output (with
spec versions, install method) stays in the subcommand for users who
want detail.

**Tests.** Extend `tests/test_version.py`:
- `cli --version` exits 0 with `cortex <version>` on stdout.
- `cli -V` exits 0 with same.
- `cli version` (subcommand) still works and produces the multi-line
  output.

### 3. `cortex refresh-state` Sources detects the project's dep-manifest

**Problem.** The Sources line currently hardcodes `pyproject.toml`:
```
- pyproject.toml + cortex package version: unknown / 0.5.0
```
On autumn-mail (a Swift project), this reads `unknown / 0.5.0` because
no `pyproject.toml` exists. The refresh-state command is for any
project, not Python projects.

**Fix.** Detect the project's primary dep-manifest by checking for the
following in the project root (first match wins, in this order):

| Manifest | Ecosystem |
|---|---|
| `pyproject.toml` | Python |
| `package.json` | Node |
| `Cargo.toml` | Rust |
| `go.mod` | Go |
| `Gemfile` | Ruby |
| `Package.swift` | Swift |
| `pom.xml` | Java/Maven |
| `build.gradle` / `build.gradle.kts` | Gradle |

Render the line as:
```
- <manifest-filename>: <project-name-or-version-extract> + cortex package version: <cortex-version>
```

For each ecosystem, extract the most-likely "version-or-name" line as
best you can without taking on full parser dependencies:
- Python: regex for `^version = "..."` then `^name = "..."`
- Node: regex for `"version": "..."` then `"name": "..."` from package.json
- Rust: same as Python (Cargo.toml uses TOML)
- Go: regex for `^module ` line in go.mod
- Ruby: skip — Gemfiles don't carry name/version; just emit the
  manifest filename
- Swift: regex for `name: "..."` in Package.swift's `Package(`
- Java: leave at filename only; pom.xml parsing isn't worth pulling
  XML for the marginal value
- Gradle: same — just filename

If NO manifest is found at all, render `- (no project manifest
detected) + cortex package version: <version>`.

**Tests.** Extend `tests/test_refresh_state.py`:
- Python project (`pyproject.toml` present) — Sources line names it.
- Swift project (`Package.swift` only) — Sources line names it.
- Empty project (no manifest) — Sources line says "no project
  manifest detected".
- Multi-manifest preference — Python wins over Node when both
  present (verifies first-match ordering).

## Implementation notes

- Put the manifest detection in a small helper —
  `src/cortex/manifests.py` (`detect_project_manifest(project_root) ->
  ManifestInfo | None`) — so the same logic is available to other
  commands later (`cortex doctor`, `cortex status`, etc.) without
  duplication. Pre-implementation checklist #1.
- Don't add a new dep for parsing TOML/JSON; stdlib `tomllib` and
  `json` are fine. Don't pull in YAML or XML parsers.

## Acceptance criteria

- `cortex grep "" --frontmatter Type:decision` against autumn-mail-
  shaped fixture (a `.cortex/templates/` and a single journal entry)
  returns only the journal entry, not the template.
- `cortex --version` and `cortex -V` print the version on one line.
- `cortex refresh-state` against a Swift-only fixture reports
  `Package.swift` in Sources, not `pyproject.toml`.
- All test cases pass.
- `bash scripts/touchstone-run.sh validate` is green.

## Out of scope

- Lockfile parsing (e.g. uv.lock / package-lock.json / Cargo.lock).
  The Sources line is for human / agent orientation, not dependency
  resolution.
- Multiple-manifest detection ("this project has both pyproject and
  package.json"). First-match wins; if a project is hybrid, the user
  can later request a richer detect-everything mode as a follow-up.
