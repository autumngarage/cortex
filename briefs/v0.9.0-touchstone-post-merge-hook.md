# Task: Touchstone post-merge hook — auto-draft `pr-merged` Cortex journal entries (v0.9.0 first work item)

You are an autonomous engineering agent dispatched to implement the Touchstone-side
of Cortex's v0.9.0 dogfood gate first work item. **This task lives in the
`autumngarage/touchstone` repo at `~/repos/touchstone`, NOT in cortex.**

You will operate in `~/repos/touchstone`, follow Touchstone's git workflow,
and ship via Touchstone's own `scripts/open-pr.sh --auto-merge` (or
equivalent). The cortex repo is read-only context for you — you may
reference its templates and CLI surface, but you do not edit cortex files.

## Goal

Add a Touchstone-managed post-merge git hook that, when a PR merges to the
default branch, shells out to `cortex journal draft pr-merged --no-edit`
to auto-create a `Type: pr-merged` Journal entry in the project's
`.cortex/journal/`. This closes Cortex Protocol Tier-1 trigger T1.9
("Pull request merged to the default branch") for any project that has
both Touchstone and Cortex installed.

This is the first v0.9.0 dogfood-gate work item. **Sequence:** ship the
Touchstone PR, enable the hook on cortex's own repo as the canary,
accumulate ≥3 auto-drafted `pr-merged` entries on cortex itself before
opening the three install PRs (conductor + touchstone + vesper). The
canary protects against the hook itself being broken.

## Required behavior

### Detection

Hook activates iff ALL of:
- Push target is the default branch (`main` or `master`).
- The repo has a `.cortex/` directory at the same level as `.git/`.
- The `cortex` CLI is on `$PATH`.
- The project's `.touchstone-config` has `[integrations.cortex]
  pr_merged_hook = "auto"` (default for newly-bootstrapped projects) or
  `"on"`. Value `"off"` disables.

If any of those is false, the hook silently exits 0. **No noisy "skipped"
output for the inactive paths** — Touchstone hooks must not pollute the
push log when they have nothing to do. Touchstone's existing hook patterns
(see `templates/`) are the precedent.

### Action

When activated:

1. Determine the merged-PR number. The hook runs server-side post-merge
   in CI OR client-side post-`merge-pr.sh` invocation. For the client
   path, the PR number is in `merge-pr.sh`'s output and can be passed via
   env (e.g. `TOUCHSTONE_MERGED_PR=N`). For the CI path, GITHUB_PR_NUMBER
   or similar. Use the most reliable path available; degrade gracefully
   if no PR number can be determined (just call `cortex journal draft
   pr-merged --no-edit` without `--pr` and let cortex pick up context
   from `gh pr view` of the most-recently-merged PR by HEAD sha).
2. Shell out: `cortex journal draft pr-merged --no-edit` (with `--pr N`
   if known). The `--no-edit` flag is critical — the hook is non-
   interactive. Cortex's `journal draft` already handles the rest
   (template fill, `git log` + `gh pr view` context, atomic write).
3. Capture stdout (the new journal entry path).
4. Append a `git add` for the new file + commit it with a message like
   `"docs(journal): auto-draft pr-merged entry for #${PR}"`. **The hook
   creates ONE follow-up commit on the default branch with just this
   journal entry.** Use `--no-verify` on this commit (safe — it's a
   well-defined automated commit, not a human-bypass) so this commit
   doesn't re-trigger the hook recursively.
5. Push the commit. If push fails, do not retry — log to stderr and exit
   1 (a real failure surfaces as visible output; an undocumented retry
   loop is worse).

### Failure modes (no silent failures)

- If `cortex journal draft` exits non-zero, capture stderr and surface to
  hook's stderr. Exit 1.
- If `cortex` is missing from PATH despite the detection passing (race
  condition), log clearly and exit 0 (degrade gracefully — don't fail the
  push because cortex was uninstalled mid-flow).
- If the resulting commit + push fails, log + exit 1.
- All three failure modes route to stderr, never silent.

### Project-side opt-out

The default for newly-bootstrapped Touchstone projects is `auto`. Existing
projects upgrading Touchstone get `auto` if they have `.cortex/` and the
`cortex` CLI when they run `touchstone update`. A project that explicitly
sets `off` in their `.touchstone-config` keeps it off through future
updates (Touchstone respects local config overrides).
