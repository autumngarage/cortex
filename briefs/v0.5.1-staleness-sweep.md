# Task: v0.5.1 staleness sweep — fix stale strings the v0.5.0 release missed

You are an autonomous engineering agent dispatched by Cortex to address
findings from the autumn-mail dogfood of v0.5.0 (2026-04-28). You operate
inside a fresh git worktree branched off `origin/main`. You will own
the fix end-to-end and ship via `scripts/open-pr.sh --auto-merge`.

## Goal

The v0.5.0 release shipped functioning code but left stale strings in
init scaffolding, status output, --help text, and the SPEC_VERSION
file template that announce shipped features as future work. Every
fresh `cortex init` today writes `.cortex/SPEC_VERSION = "0.5.0-dev"`
and a state.md scaffold that says *"`cortex refresh-state` ships in
v0.4.0"* — the command shipped in v0.5.0 (and was in v0.4.0). Fix all
of them in one polish PR.

## Required fixes

Each fix is a string change. No new behavior. Address them all in one
PR; ship as v0.5.1 patch release.

1. **`.cortex/SPEC_VERSION` written by `cortex init` — `0.5.0-dev` →
   `0.5.0`.** Find the hardcoded version in the init scaffold (likely
   `src/cortex/commands/init.py` or `src/cortex/_data/`) and bump.
   The string should match `cortex.SPEC_VERSION_FILE_VALUE` (or
   whatever the source-of-truth name is) — if it's currently a string
   literal, define a single constant and reference it.

2. **`cortex init --help` description string** — currently
   `"Scaffold a SPEC-v0.4.0-dev-conformant .cortex/ directory in the
   target project."` Update to `SPEC-v0.5.0`. Find via
   `grep -rn "v0.4.0-dev-conformant"`.

3. **Init scaffold `state.md` body prose** — the placeholder text says
   *"When the deterministic `cortex refresh-state` command ships in
   v0.4.0..."* Strip the "ships in v0.4.0" phrasing entirely; the
   command is shipped. Replace with present-tense guidance pointing
   at how `cortex refresh-state` regenerates this layer when run.

4. **Init scaffold `state.md` `Incomplete:` line** — currently reads
   *"All sources — listed by `cortex init` from a project scan;
   `cortex refresh-state` will synthesize them in v0.4.0."* Change to
   present-tense: *"All sources — listed by `cortex init` from a
   project scan; run `cortex refresh-state` to regenerate this layer
   from primary sources."*

5. **`cortex status` promotion-queue absent message** — currently
   reads *"not yet initialised (`.cortex/.index.json` absent;
   populated by the v0.6.0 lifecycle commands)"*. The index writer
   shipped in v0.5.0. Change to *"not yet initialised
   (`.cortex/.index.json` absent; run `cortex refresh-index` to
   populate)"*. Find via `grep -rn "v0.6.0 lifecycle"`.

6. **`cortex init` post-scaffold message** — `Scaffolded /path/.cortex
   (spec v0.5.0-dev).` should report `(spec v0.5.0)`. Same root cause
   as #1.

7. **`.cortex/SPEC_VERSION` in this Cortex repo** — already at `0.5.0`
   (fixed in PR #66). Verify it's still `0.5.0` and not regressed.

8. **Sweep for any other `0.4.0-dev` or `0.5.0-dev` string in source
   code** that should be `0.5.0`. `grep -rn "0\.4\.0-dev\|0\.5\.0-dev"
   src/ tests/ .cortex/templates/` and update each to match present
   reality. Tests that intentionally pin to a SPEC-version-string
   value are fine if they're checking version-bump behavior; use
   judgment.

## Test additions / changes

- Add `tests/test_init_spec_version.py` (or extend existing init
  tests) asserting `cortex init --yes` writes `.cortex/SPEC_VERSION`
  with content equal to `cortex.__init__.SPEC_VERSION_LITERAL` (or
  the canonical constant) — never a `-dev` suffix when the package
  version doesn't carry one.
- Existing init-scan tests should still pass without modification.

## Acceptance criteria

- `grep -rn "0\.4\.0-dev\|0\.5\.0-dev" src/ tests/ .cortex/templates/`
  returns no hits except in tests that intentionally fix version
  strings (those should have a comment explaining why).
- A fresh `cortex init --yes` against a temp dir writes `0.5.0` (not
  `0.5.0-dev`) to `.cortex/SPEC_VERSION`.
- `cortex init --help` first line says `SPEC-v0.5.0-conformant`.
- `cortex status` against an empty `.cortex/` (no `.index.json`)
  mentions `cortex refresh-index`, not `v0.6.0 lifecycle commands`.
- Init scaffold `state.md` body has no future-tense "ships in vX.Y.Z"
  references for commands that have shipped.
- `bash scripts/touchstone-run.sh validate` is green.

## How to ship

```sh
bash scripts/touchstone-run.sh validate
git add -A   # acceptable here — purely string-update changes
git commit -m "fix: v0.5.1 staleness sweep — strings the v0.5.0 release missed"
bash scripts/open-pr.sh --auto-merge
```

The release tag (v0.5.1) follows in a separate step after this PR
plus the cortex-next placeholder filter PR plus the polish PR all
land. Do NOT bump version or tag; the release script handles that
once all three v0.5.1 PRs are merged.
