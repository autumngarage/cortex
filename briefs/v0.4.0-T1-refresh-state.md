# Task: Implement deterministic `cortex refresh-state`

You are an autonomous engineering agent dispatched by Cortex to implement Tier 2
(v0.4.0) work item T1 from `.cortex/plans/cortex-v1.md`. You operate inside a
fresh git worktree branched off `origin/main`. You have read/write/exec access
to the worktree only. You will own the implementation end-to-end and ship it
via `scripts/open-pr.sh --auto-merge`.

## Goal

Implement `cortex refresh-state`: a deterministic command that regenerates
`.cortex/state.md` by walking primary sources (plans, journal, doctrine,
templates, etc.) and rendering a SPEC-conformant file with hand-authored
regions preserved verbatim through regeneration.

This is the **Tier 2 inflection point** that makes a project's state file stop
drifting. After this command exists and is used, `state.md` becomes a derived
artifact that always reflects current reality — not a hand-edited document
that quietly goes stale.

## Required behavior

1. **Seven-field header regenerated** per SPEC § 4.3:
   - `Generated: <ISO-8601 timestamp with timezone>`
   - `Generator: cortex refresh-state v<__version__>`
   - `Sources:` — list of input artifacts walked (HEAD sha, plans count,
     journal entry count + date range, doctrine count, templates count, case
     studies count, SPEC version, pyproject + cortex package version)
   - `Corpus:` — one-line summary
   - `Omitted:` — explicit list of intentionally-excluded inputs (e.g.,
     `.cortex/.index.json` if absent)
   - `Incomplete:` — non-empty list when a source could not be fully read
   - `Conflicts-preserved:` — pairs of contradictory journal claims; empty
     list `[]` when none
   - `Spec:` — the spec version supported by this regeneration

2. **Auto-walked sections.** Three sections are entirely regenerated from
   primary sources. Their contents are computed; the user does not hand-edit:
   - `## Active plans` — bullets per active plan (slug, Goal, completion %)
   - `## Shipped recently` — chronological bullets pulled from journal
     entries with `Type: release` or `Type: pr-merged` or
     `Type: plan-transition` whose status moved to `shipped`
   - `## Stale-now / handle-later` — derived from `cortex plan status`
     staleness flags (active plans whose last `Updated-by` is older than
     14 days AND have open checkboxes) plus journal entries tagged with
     `Stale-by:` frontmatter

3. **Hand-authored regions preserved verbatim.** Sections (or sub-regions)
   between `<!-- cortex:hand -->` and `<!-- cortex:end-hand -->` markers
   survive regeneration unchanged, byte-for-byte. The marker convention is
   **per-section pairs** (one open + one close per intended hand-authored
   region); multiple pairs can coexist with regenerated regions in the same
   file. The current `## Current work` section in this repo's `state.md`
   should be wrapped in markers as part of this PR so it survives the first
   regeneration.

4. **Idempotent.** `cortex refresh-state` followed immediately by another
   `cortex refresh-state` produces a byte-identical `state.md` (modulo the
   `Generated:` timestamp — the implementation MUST normalize the timestamp
   to a deterministic value when running under `CORTEX_DETERMINISTIC=1` so
   the idempotency test can assert byte-for-byte equality across two runs).

5. **`--dry-run`** flag prints the would-be `state.md` to stdout without
   writing.

6. **`--path <DIR>`** runs against an arbitrary `.cortex/`-bearing project,
   matching the convention of `cortex status --path`.

## Implementation outline

- New module: `src/cortex/commands/refresh_state.py` — Click command.
- New module: `src/cortex/state_render.py` — pure rendering logic (header
  serialization, section walkers, marker preservation). Keep this side-effect
  free so it can be unit-tested without filesystem mocks.
- Reuse: `src/cortex/frontmatter.py`, `src/cortex/plans.py` (extracting plan
  parsing into this module if it is currently inline elsewhere), the existing
  manifest source-walking helpers in `src/cortex/manifest.py` for the seven-
  field header materials.
- Wire the new command into `src/cortex/cli.py` (or wherever the command
  registry lives — grep for the existing `cortex.commands.journal` and
  `cortex.commands.plan` registrations).
- The `CORTEX_DETERMINISTIC=1` environment variable freezes `Generated:` to a
  constant ISO-8601 string `2000-01-01T00:00:00+00:00` and any non-determin-
  istic ordering to lexicographic. This is for tests only; document it in the
  command's docstring.

## Tests required

Place tests under `tests/test_refresh_state.py`. Use real `tmp_path` filesystem
fixtures (no mocked subprocess, no mocked filesystem) — match the convention
established in `tests/test_doctor.py` and `tests/test_journal.py`.

Required test cases:

1. **Idempotency.** Run `refresh-state` twice on a fixture `.cortex/`; assert
   the second output is byte-identical to the first (under
   `CORTEX_DETERMINISTIC=1`).
2. **Marker preservation.** Fixture `state.md` contains a hand-authored
   region between markers. Mutate the body inside the markers (e.g., add a
   line). Run `refresh-state`. Assert the mutation is preserved verbatim.
3. **Multiple marker pairs.** Fixture has two separate hand-authored regions
   (e.g., `## Current work` and `## Open questions`). Both survive.
4. **Auto-walked section contents.** Fixture has two active plans, one
   shipped recently, one stale. Assert `## Active plans` lists both active
   plans with their Goal-hash and computed completion %; `## Shipped
   recently` includes the shipped one; `## Stale-now / handle-later` lists
   the stale one.
5. **Seven-field header completeness.** All seven fields present with the
   correct shape; `Omitted:` lists `.cortex/.index.json` when absent;
   `Incomplete:` is `[]` on a clean fixture.
6. **`--dry-run`.** Does not modify the filesystem; output goes to stdout.
7. **`--path`.** Targets an arbitrary fixture directory.

## Engineering principles to apply

Read `principles/engineering-principles.md`. Especially relevant:
- **No silent failures** — surface every parse error with file path + reason.
- **One code path** — no mode flags forking the renderer; deterministic mode
  is an env var that affects the timestamp formatter only.
- **Derive, don't persist** — state.md is derived from plans + journal +
  doctrine + templates + case studies; no separate cache file.
- **Every fix gets a test** — every behavior described in this brief has a
  test in `tests/test_refresh_state.py`.

## How to ship

When the implementation is done and tests pass:

```sh
bash scripts/touchstone-run.sh validate     # full local validation
git status                                   # stage explicit paths
git add src/cortex/commands/refresh_state.py src/cortex/state_render.py \
        src/cortex/cli.py tests/test_refresh_state.py .cortex/state.md \
        .cortex/journal/<your-decision-entry>.md
git commit -m "feat: cortex refresh-state — deterministic state regeneration"
bash scripts/open-pr.sh --auto-merge
```

`open-pr.sh --auto-merge` will push the branch, open the PR, run Codex review,
auto-fix low-risk findings, squash-merge, and sync main. If Codex blocks on a
high-scrutiny finding, address it in a follow-up commit and push again — the
hook re-runs.

If anything is unclear in this brief, prefer the SPEC references over guessing.
If the SPEC is ambiguous, flag it in the PR description as an open question
and pick the most conservative interpretation.
