### Index location and lifecycle

- Index path: `.cortex/.index/chunks.sqlite` (gitignored â€” the existing
  `.cortex/.gitignore` already covers `.index/` per cortex-retrieve.md;
  verify and add if missing).
- Index format: SQLite with FTS5 extension (built into Python's stdlib
  `sqlite3` since 3.7+; no new dep needed for S1).
- Schema (S1 version â€” S2 will add a `vectors` table alongside):
  ```sql
  CREATE TABLE chunks (
    id INTEGER PRIMARY KEY,
    path TEXT NOT NULL,        -- relative to .cortex/
    chunk_idx INTEGER NOT NULL,-- 0-indexed within file
    start_line INTEGER NOT NULL,
    end_line INTEGER NOT NULL,
    content TEXT NOT NULL,
    frontmatter_json TEXT,     -- serialized frontmatter for the file (NULL if none)
    file_mtime REAL NOT NULL,
    file_size INTEGER NOT NULL,
    UNIQUE(path, chunk_idx)
  );
  CREATE VIRTUAL TABLE chunks_fts USING fts5(
    content, path,
    content='chunks', content_rowid='id'
  );
  CREATE TABLE meta (
    key TEXT PRIMARY KEY,
    value TEXT NOT NULL
  );
  -- meta keys: 'schema_version' (int), 'cortex_version' (semver str), 'built_at' (iso8601 str)
  ```

### Chunking strategy

Per cortex-retrieve.md council delta #6: 500â€“800 token target, 100â€“150
overlap. Implementation:
- Split each markdown file at section boundaries first (`^## ` headers).
  If a section is shorter than the target, keep it whole. If longer, split
  on paragraph boundaries (`\n\n`) until each chunk fits.
- Token estimate: use a simple word-count Ă— 1.3 heuristic (no tiktoken
  dep for S1). When fastembed lands in S2, switch to its tokenizer.
- Overlap: when a section is split, each non-final chunk includes the
  next section's first paragraph as overlap context.
- Chunks track `start_line` + `end_line` for citation back to the source.
- `frontmatter_json` is the parsed YAML frontmatter of the source file
  (as a JSON string), shared across all chunks of that file. NULL if the
  file has no frontmatter.

### Index rebuild + invalidation

**Per council delta #1 â€” invalidation MUST use mtime+size against the
working tree, NOT `git ls-tree`.** Uncommitted edits to journal files must
be visible in retrieval results within one rebuild cycle. Rebuild logic:

1. Walk `.cortex/` for `*.md` files (exclude `.cortex/.index/`,
   `.cortex/.gitignore`, hidden files).
2. For each file: read mtime + size from filesystem. If `(path, mtime, size)`
   matches a row in the `chunks` table, file is unchanged â†’ skip.
3. If file is new or changed: re-chunk, replace all rows for that path, FTS5
   rebuilds automatically via the contentless-table mechanism.
4. If a path exists in `chunks` but no longer on disk: delete its rows.
5. Update `meta.built_at` to ISO8601 now.

**Auto-rebuild trigger:** when `cortex retrieve` runs, check the staleness
of the index against the working tree. If any markdown file under `.cortex/`
has changed (mtime or size) since `meta.built_at`, rebuild incrementally
before answering the query â€” unless `--no-rebuild` is set, in which case
emit a one-line stderr notice that results may be stale.

**Atomic write:** new index built at `.cortex/.index/chunks.sqlite.tmp`;
on success, atomic rename to `.cortex/.index/chunks.sqlite`. Prevents
torn writes if interrupted mid-rebuild.

### `cortex refresh-index` integration

The existing `cortex refresh-index` command writes `.cortex/.index.json`
(promotion-queue index). Add a new flag `--retrieve` that ALSO rebuilds the
chunks index. Default behavior is unchanged (only `--retrieve` triggers
the chunks-index rebuild). When `--retrieve` is set, both indexes rebuild.

Auto-trigger from existing commands: after `cortex journal draft` writes
a new entry AND after `cortex refresh-state` runs, IF `.cortex/.index/chunks.sqlite`
exists, trigger an incremental rebuild of the chunks index. (If the
chunks index doesn't exist, do nothing â€” the user hasn't opted into
retrieve yet.)

### `cortex grep` is untouched

Per council delta #2, this is non-negotiable. `cortex grep` continues to
work zero-deps, never imports the new retrieve module, never touches the
sqlite index. Both commands ship side-by-side.

### Bare-repo / missing-runtime fallback

S1 uses only stdlib `sqlite3` + FTS5 â€” no native deps to fail. But if for
some reason `sqlite3` lacks FTS5 (extremely rare; some minimal distros
build sqlite without FTS5), `cortex retrieve` must:
1. Detect the missing extension on first run (try a `CREATE VIRTUAL TABLE
   ... USING fts5` and catch the operational error).
2. Print a clear actionable message: `"FTS5 extension not available in this
   sqlite3 build; cortex retrieve requires FTS5. Falling back to ripgrep
   for this query. Install a sqlite3 build with FTS5 or use 'cortex grep'
   directly."`
3. Shell out to `cortex grep` with the same query and emit those results.
4. Exit 0 (graceful degradation â€” never silent, but not a crash).
