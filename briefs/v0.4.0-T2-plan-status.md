# Task: Implement `cortex plan status`

You are an autonomous engineering agent dispatched by Cortex to implement
Tier 2 (v0.4.0) work item T2 from `.cortex/plans/cortex-v1.md`. You operate
inside a fresh git worktree branched off `origin/main`. You will own the
implementation end-to-end and ship it via `scripts/open-pr.sh --auto-merge`.

## Goal

Implement `cortex plan status` — a command that walks `.cortex/plans/*.md`,
computes per-plan completion percent and staleness, and prints a structured
report. Used directly by `cortex next` (separate task) and by `cortex doctor`
(when it expands its invariant checks in Tier 4).

## Required behavior

For every `.cortex/plans/*.md` file (excluding `_archived/`, `template.md`):

1. Parse frontmatter (`Status`, `Written`, `Author`, `Goal-hash`, `Updated-by`,
   `Cites`) per SPEC § 3.4.
2. Parse `## Work items` body and count checkbox states (`- [ ]` vs `- [x]`
   vs `- [~]`). Sub-section headers (`### ...`) are bucketed but do not
   themselves count.
3. Compute completion percent: `done / (done + open)`. `[~]` (in-progress)
   counts as half-done for the percentage, fully-closed for the open-count
   used by staleness.
4. Determine staleness: a plan is **stale** when:
   - `Status: active` AND
   - the trailing entry in `Updated-by:` is older than 14 days from today AND
   - at least one checkbox is `[ ]` (open).
5. Emit per-plan report. Default human-readable output:
   ```
   plans/<slug>.md
     Status:      <status>
     Goal-hash:   <hash>
     Completion:  N% (X of Y items)
     Last update: <iso-date> (<N days ago>)
     Stale:       yes|no
   ```
6. `--json` emits a machine-readable list of objects.
7. `--stale-only` prints only plans flagged stale.
8. `--path <DIR>` targets an arbitrary `.cortex/`-bearing project.
9. Exit code 0 always (informational); a `--strict` flag is parked for now —
   not in scope for this PR.

## Implementation outline

- New module: `src/cortex/commands/plan_status.py` — Click sub-command of
  the existing `cortex plan` group (created by `cortex plan spawn`).
- Move shared plan parsing into `src/cortex/plans.py` if it is not already
  consolidated there (grep for inline parsing in `src/cortex/commands/`).
- Reuse `src/cortex/frontmatter.py` for YAML frontmatter parsing.
- The 14-day threshold is a named constant `STALENESS_DAYS = 14` in
  `src/cortex/plans.py` with a docstring citing SPEC § 4.7's promotion-queue
  aging threshold (the same threshold is reused intentionally).

## Tests required

Place tests under `tests/test_plan_status.py`. Use real `tmp_path` fixtures.

Required test cases (per the parent plan's "Tests" line in v0.4.0 T2):

1. **Zero work-items.** Plan with no `## Work items` body — completion %
   reports 0/0, no division-by-zero error, staleness false.
2. **All-complete.** Plan with all `[x]` — 100%, not stale (no open items).
3. **Mixed.** Plan with 3 done + 2 open + 1 in-progress — completion = 70%
   (3 + 0.5 = 3.5 of 5 items: open count for staleness uses 2).
4. **Stale-active.** Plan with `Status: active`, last `Updated-by` 30 days
   ago, open checkbox — flagged stale.
5. **Fresh-active.** Plan with `Status: active`, last `Updated-by` today,
   open checkbox — not stale.
6. **Shipped.** Plan with `Status: shipped` — never stale regardless of
   age (only active plans flag stale).
7. **Cancelled.** Plan with `Status: cancelled` — never stale.
8. **--json output shape.** Asserted against a fixture-derived expected
   structure.
9. **--stale-only filters correctly.**
