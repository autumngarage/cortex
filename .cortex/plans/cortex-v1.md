## Why (grounding)

The session-pickup gap is what Cortex exists to close ([`doctrine/0001-why-cortex-exists`](../doctrine/0001-why-cortex-exists.md)). Repos are starved of context — plans, changelogs, why-this-exists rationale — and the cost of that starvation compounds: agents make confident decisions against stale inputs, humans pick up codebases without the load-bearing reasoning, and the project's memory lives in commit messages nobody reads.

The conductor case study ([`docs/case-studies/2026-04-24-stale-claude-md-steered-agent-wrong.md`](../../docs/case-studies/2026-04-24-stale-claude-md-steered-agent-wrong.md)) is the design test case: a stale `CLAUDE.md` claim ("tap planned for v0.1.0; not yet wired") confidently steered an agent to recommend the wrong install path eight releases after the tap had shipped. v1.0 is done when that *class* of incident is structurally impossible — the trust layer (`--audit-instructions`, `Verified:`) catches stale external-artifact claims, the write layer (`cortex journal draft`, T1.10) records when reality changed, the lifecycle layer (`promote`, `index`) graduates load-bearing observations to durable doctrine, and a sustained period of real work on three live projects exercises every layer against three different content shapes.

## Approach

**Feature-readiness, not calendar.** The path to v1.0 is sequenced by what a real repo can rely on at each tier — not by date. Each tier corresponds to one or more minor releases, but the *commitment* is the capability bar, not the version number. Tier 2 + Tier 3 telescoped into v0.5.0 (released 2026-04-28) because the work landed faster than originally scoped; bug-fix point releases (v0.5.x) ship freely within a tier when issues surface, as v0.5.1 demonstrated against the autumn-mail one-off Swift dogfood.

**The "automatic" inflection point sits inside v0.9.0.** Tier 3's Cortex-side surface (`--audit-instructions`, `Verified:`) shipped at v0.5.0, but "install and walk away" only becomes a real claim once the Touchstone post-merge hook validates under real merge load — which is exactly what the v0.9.0 dogfood gate is for. The hook canary-tests on this repo first, then runs across the three targets during the gate.

**Critical-path framing.** Every v1.0 work item answers yes to: *does this make Cortex usable on a real project today?* Items that don't (LLM polish, triad-mode infra, low-frequency UX, MCP transport, retention automation as cleanup not visibility) move to `## Follow-ups (deferred)` with explicit revisit conditions. The bar is not "would this be valuable" — almost everything would; the bar is "does production-on-a-real-project depend on it."

**There is no Tier 5.** All deferred items are v1.x candidates and will not be reconsidered until v1.0 ships. Naming them as a tier would signal critical path, which they are not.

**Deterministic only on the v1.0 path.** Per [`doctrine/0005-scope-boundaries-v2`](../doctrine/0005-scope-boundaries-v2.md) #7, LLM features were already opt-in `--enhance` flags. The v0.3.0 → v1.0 sequence takes this further: LLM commands (`refresh-map`, `refresh-state --enhance`, `cortex next --enhance`) defer to v1.x because the conductor case study evidence is that prose polish *hides* staleness — synthesis over a stale base would re-create the conductor incident inside Cortex itself. Trust-layer features (`--audit-instructions`, `Verified:`) ship first; LLM polish gets reconsidered only after the trust layer has dogfooded across three targets.

**Three-target dogfood with overlapping active use, not literal-same-day.** Three different content shapes, three different edit cadences, two different ecosystems (Python + Swift). **conductor** is the case-study origin and validates `--audit-instructions` against a project that has demonstrably suffered the failure mode. **touchstone** is the foundation tool and validates Cortex stays out of Touchstone-managed write paths in the project that defines those paths. **vesper** is Swift, not Python, and validates language-agnosticism. autumn-mail (Swift) was a one-off readiness check — not the named gate target — but it proved the dogfood-driven-patch loop scales via conductor dispatch and de-risks vesper as an ecosystem-coverage proxy.

**Dogfood gate is the engineering release; v1.0 is ceremony plus compatibility.** If the three-target dogfood surfaces structural bugs, point releases (v0.9.x) ship before v1.0 declares freeze. v1.0 itself adds three things beyond ceremony: pre-1.0 compatibility audit (does an old `.cortex/` still work or do users need a migration path?), `.cortex/config.toml` schema reference (a user-facing config format without published schema is an adoption wall), and SPEC-to-test traceability matrix (per Doctrine 0003 — if SPEC is the primary deliverable, every normative § needs a check).

**Codesight cross-pollination integrated, then sharpened by council review.** Three codesight ideas land in the v1.0 path; two park as deferred follow-ups; one was demoted from a gate item to opportunistic evidence after council review. The integrations are token-cost measurement (opportunistic, no public-ratio commitment), inline ADR absorption via existing `cortex init` scan-and-absorb (no separate gate work), and the related-work paragraph in PITCH/PRIOR_ART. The deferred items — an MCP transport surface and a standalone `cortex import-knowledge` command — have explicit revisit conditions tied to dogfood-evidence demand. See [`journal/2026-04-28-codesight-cross-pollination-and-council-review`](../journal/2026-04-28-codesight-cross-pollination-and-council-review.md) for the council deltas in full.

### v0.7.0 — Retrieval interface (`cortex retrieve`) — 🟡 S1 shipped at v0.7.0; S2 + S3 in flight

Goal: ship `cortex retrieve` as a real CLI so projects with >100 entries get queries that work on terminology drift, not just exact-match grep. Standalone-first: cortex must work end-to-end without Sentinel. Sentinel-ready: the `--json` shape is the contract Sentinel's Planner consumes in a real cycle. **Sub-plan:** [`plans/cortex-retrieve.md`](./cortex-retrieve.md) (S0–S3 slice plan; Status: active).

Why this is on the launch path (added 2026-05-02): recency-by-grep is sufficient for early projects but breaks past ~100 entries — agents miss the entry that *would* answer their question because grep needs the exact term. Cortex's promise is "your project's memory survives." Failing on terminology drift breaks the promise. Doctrine 0006 (`scope-boundaries-v3`, shipped 2026-04-29) already narrows the prior "not a vector store" framing to allow Cortex to own a retrieval interface over its own format; v0.7.0 implements that doctrine.

Why not deferred: previously parked alongside the `--enhance` LLM polish family, but that conflated two different things. LLM polish is genuinely premature (it would polish stale facts). Local-CPU embeddings are deterministic, network-free, and the index is gitignored derived state — none of the "polish hides staleness" failure mode applies. Retrieval upgrades the read surface without touching the storage source of truth.
