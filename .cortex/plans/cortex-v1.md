# Ship Cortex v1.0

> The one plan from here to v1.0. Sequenced as **feature-readiness tiers**, not calendar releases. Each tier names a capability bar a real project can rely on. The forcing function: install Cortex on three real projects (conductor + touchstone + vesper), do real work, no surprises.

## Why (grounding)

The session-pickup gap is what Cortex exists to close ([`doctrine/0001-why-cortex-exists`](../doctrine/0001-why-cortex-exists.md)). Repos are starved of context — plans, changelogs, why-this-exists rationale — and the cost of that starvation compounds: agents make confident decisions against stale inputs, humans pick up codebases without the load-bearing reasoning, and the project's memory lives in commit messages nobody reads.

The conductor case study ([`docs/case-studies/2026-04-24-stale-claude-md-steered-agent-wrong.md`](../../docs/case-studies/2026-04-24-stale-claude-md-steered-agent-wrong.md)) is the design test case: a stale `CLAUDE.md` claim ("tap planned for v0.1.0; not yet wired") confidently steered an agent to recommend the wrong install path eight releases after the tap had shipped. v1.0 is done when that *class* of incident is structurally impossible — the trust layer (`--audit-instructions`, `Verified:`) catches stale external-artifact claims, the write layer (`cortex journal draft`, T1.10) records when reality changed, the lifecycle layer (`promote`, `index`) graduates load-bearing observations to durable doctrine, and a sustained period of real work on three live projects exercises every layer against three different content shapes.

## Success Criteria

This plan is done when Cortex v1.0 ships into real-project use on three install targets simultaneously, with each target satisfying its tier-by-tier exit bars. Measurable per tier:

1. **Tier 1 — Templated, manual (shipped).** `cortex init` scaffolds cleanly on a fresh repo; `cortex journal draft <type>` produces a SPEC-conformant entry pre-filled from `git log` + `gh pr view`; `cortex plan spawn <slug>` scaffolds a SPEC-conformant active Plan; `cortex doctor --audit` walks `git log` and warns when expected entries are missing; orphan-deferral check passes on this repo with no waivers.
2. **Tier 2 — State self-maintains.** `cortex refresh-state` byte-identical on unchanged inputs (idempotency test) with `<!-- cortex:hand -->` markers preserved verbatim through regeneration; `cortex next` produces a non-empty ranked list on this repo with stable citations; `cortex plan status` flags any active plan staler than 14 days.
3. **Tier 3 — Automatic on merge.** A sustained-work period on this repo produces ≥ 5 auto-drafted `pr-merged` entries via the Touchstone post-merge hook; `cortex doctor --audit-instructions` on this repo produces clean exit, on each install target produces unambiguous output (either non-trivial findings, or "checked N claims, all verified" — never silent); manifest output surfaces stale `Verified:` timestamps inline.
4. **Tier 4 — Lifecycle complete.** `.cortex/.index.json` writer produces a populated index on this repo; end-to-end promotion (journal observation → doctrine entry via `cortex promote <id>`) succeeds with `Type: promotion` Journal entry written; remaining doctor invariant checks (append-only, immutable-Doctrine, promotion-queue, single-authority drift, T1.4, claim-trace) pass on this repo.
5. **Three-target dogfood gate.** Cortex installed on **conductor**, **touchstone**, and **vesper** simultaneously; sustained period of real work using `cortex journal draft` for new entries on each; `cortex doctor --audit-instructions` exercised against each target's external-artifact claims; ≥ 1 case-study-style journal entry per target capturing surfaced friction; zero crashes on any target; Cortex stays out of Touchstone-managed write paths on every target; user assessment "I'd rather use this than hand-write" on each target.
6. **v1.0 production release.** `SPEC.md` frozen (no `-dev` suffix; bumped to 1.0.0); README / PITCH refreshed to tell the production-ready story grounded in the three-target dogfood; Homebrew formula updated; GitHub Release covering the full Tier-1 → v1.0 arc.

## Approach

**Feature-readiness, not calendar.** The path to v1.0 is sequenced by what a real repo can rely on at each tier — not by date. Each tier corresponds to a minor release (Tier 2 = v0.4.0, Tier 3 = v0.5.0, Tier 4 = v0.6.0), but the *commitment* is the capability bar, not the version number. Bug-fix point releases (v0.4.x, v0.5.x) ship freely within a tier when issues surface.

**The "automatic" inflection point is Tier 3.** Tier 1 (today) is templated-manual: useful but the user must remember to invoke commands. Tier 2 makes state self-maintaining (read-side stops drifting). Tier 3 is when "install and walk away" becomes real — the Touchstone post-merge hook fires `cortex journal draft pr-merged --no-edit` on every merge with no user action, and `--audit-instructions` actively catches stale CLAUDE.md / README claims before they steer an agent wrong. **For repos starved of context that need to self-update, Tier 3 is the install moment.**

**Critical-path framing.** Every v1.0 work item answers yes to: *does this make Cortex usable on a real project today?* Items that don't (LLM polish, triad-mode infra, low-frequency UX) move to `## Follow-ups (deferred)` with explicit revisit conditions per [`journal/2026-04-24-production-release-rerank`](../journal/2026-04-24-production-release-rerank.md). The bar is not "would this be valuable" — almost everything would; the bar is "does production-on-a-real-project depend on it."

**Deterministic only on the v1.0 path.** Per [`doctrine/0005-scope-boundaries-v2`](../doctrine/0005-scope-boundaries-v2.md) #7, LLM features were already opt-in `--enhance` flags. The v0.3.0 → v1.0 sequence takes this further: LLM commands (`refresh-map`, `refresh-state --enhance`, `cortex next --enhance`) defer to v1.x because the conductor case study evidence is that prose polish *hides* staleness — synthesis over a stale base would re-create the conductor incident inside Cortex itself. Trust-layer features (`--audit-instructions`, `Verified:`) ship first; LLM polish gets reconsidered only after the trust layer has dogfooded.

**Three-target dogfood, not one.** The original plan named touchstone as the single dogfood target on the argument that sibling-tool composition is a strong v1.0 gate. The session-reflection rerank widens this to three: **conductor** (the case-study origin — validates `--audit-instructions` against a project that has demonstrably suffered the failure mode), **touchstone** (sibling-composition test — validates Cortex stays out of Touchstone-managed write paths on the foundation tool), and **vesper** (Swift / non-Python — validates language-agnosticism and stresses the audit against a different dep manifest). Three different content shapes, three different edit cadences, three different ecosystems. Wider evidence; better-designed `[audit-instructions]` config schema; stronger v1.0 production claim.

**Dogfood gate is the engineering release; v1.0 is ceremony.** If the three-target dogfood surfaces structural bugs, point releases (v0.9.x) ship before v1.0 declares freeze. This matches industry convention (1.0 = stable + frozen, not 1.0 = first attempt) and keeps the production-readiness story honest.

### Tier 2 — State self-maintains (v0.4.0)

Goal: state.md stays current automatically and the user gets a deterministic "what to work on" command. No LLM dependency. After this tier, a repo that runs `cortex refresh-state` after each meaningful event has a state file that always derives from primary sources; drift between state.md and reality stops being a thing.

- [ ] **Deterministic `cortex refresh-state`** — seven-field header regenerated; auto-walked `## Active plans`, `## Shipped recently`, `## Stale-now / handle-later` sections from plans + journal; hand-authored regions between `<!-- cortex:hand -->` / `<!-- cortex:end-hand -->` markers survive verbatim; byte-identical output on unchanged inputs (idempotency test). Marker convention: per-section pairs (one pair per intended hand-authored region) so multiple hand-authored regions can coexist with regenerated regions in the same file. New module `src/cortex/commands/refresh_state.py` + helpers in `src/cortex/state_render.py`. Tests under `tests/test_refresh_state.py`.
- [ ] **`cortex next` deterministic MVP** (case-study item #4) — walks `.cortex/state.md` `## Current work` section + `## Open questions` + active-plan open checkboxes (reuses `cortex plan status` parser) + `docs/case-studies/*.md` newer than N days (default 30); produces a ranked list with stable citations to each source. `--json` for scripting. Deterministic priority bands: P0 (open checkboxes on active non-stale plans + state.md Current work items), P1 (state.md Open questions), P2 (recent case studies). The LLM-enhanced layer (`--enhance`) is parked at item #7 in [`journal/2026-04-24-production-release-rerank`](../journal/2026-04-24-production-release-rerank.md) and ships post-v1.0.
- [ ] **`cortex plan status`** — per-plan completion % (checkboxes parsed) + staleness flag (active plans with last `Updated-by` older than 14 days and open checkboxes). `--json` emits machine-readable output. Used directly by `cortex next` and by `cortex doctor`.
- [ ] Tests — `refresh-state` idempotency test, marker-preservation test (edited body between markers survives verbatim), `cortex next` ranked-output test against fixture state.md, `cortex plan status` against synthetic plan files (zero work-items, all-complete, mixed, stale-active, fresh-active, shipped, cancelled).
- [ ] v0.4.0 release — version bump in `src/cortex/__init__.py` + `pyproject.toml`; uv.lock regen with pinned `exclude-newer`; SPEC bump only if marker convention surfaces a § 4.3 amendment (assess at first implementation); release journal entry via `cortex journal draft release` post-tag; `gh release create v0.4.0 --generate-notes`; verify `release.yml` auto-bumps the homebrew-cortex formula.

### Tier 3 — Automatic on merge (v0.5.0)

Goal: external claims stay honest; the journal fills itself from real merge events; the trust model the conductor case study demands is in place before any LLM polish gets considered. **This tier is the inflection point — after it, "install Cortex and walk away" becomes a real claim.**

- [ ] **Touchstone post-merge hook** (in `autumngarage/touchstone`) — shells out to `cortex journal draft --type pr-merged --no-edit` on default-branch merges; opt-in per project via `.touchstone-config`. Cortex side is graceful degradation: if `cortex` not on PATH, hook is a no-op with a one-line stderr note. Touchstone PR is a separate ship; Cortex side is the `--no-edit` flag verification and any glue needed for piped-context input.
- [ ] **`cortex doctor --audit-instructions`** (case-study item #3) — scans `CLAUDE.md` / `AGENTS.md` / `README.md` for claims about external artifacts: filesystem siblings via `~/Repos/<name>` checks, Homebrew taps via `brew tap-info`, PyPI packages via `pip show` or HTTPS HEAD on the index, `gh release list` for tap/main repos, URL liveness HEAD checks. Configurable via `.cortex/config.toml` section `[audit-instructions]` naming source-of-truth artifacts per project (e.g., `homebrew_tap = "autumngarage/cortex"`, `siblings = ["~/Repos/touchstone"]`, `pypi_package = "cortex-cli"`). Warnings by default; errors under `--strict` so Touchstone pre-push (deferred — v1.x) can use it later. Two non-goals carried from the case study: does not police narrative/principles prose; does not require release-per-commit hygiene. Output never silent — even on clean projects emits "checked N claims, all verified."
- [ ] **Manifest provenance — per-fact `Verified:` per SPEC § 4.3 extension** (case-study item #5) — SPEC minor bump so derived facts inside `state.md` / `doctrine/*.md` can carry a `Verified: <date>` tag on individual bullets (not just the whole file's `Generated:` timestamp). `cortex manifest` surfaces stale `Verified:` warnings inline so agents see "this fact was last verified 180 days ago" next to the fact itself. Pairs with `--audit-instructions` (active check) as the passive freshness signal.
- [ ] Tests — `--audit-instructions` against a fixture with known-stale claims, `Verified:` parser unit tests, end-to-end test that `cortex manifest` warns inline. Touchstone-side hook tested in the touchstone repo.
- [ ] v0.5.0 release (Cortex side); separate Touchstone release.

### Three-target dogfood gate (v0.9.0)

Goal: install Cortex on **conductor**, **touchstone**, and **vesper** simultaneously; sustained period of real work; fix what surfaces. This is the engineering release-gate. v1.0.0 is ceremony; this gate is the truth.

**Why three, not one.** Wider evidence (three content shapes, three edit cadences, two ecosystems Python + Swift); better-designed `[audit-instructions]` config schema (forced to face three real CLAUDE.md surfaces); stronger v1.0 production claim. **conductor** is the case-study origin and validates `--audit-instructions` against a project that has demonstrably suffered the failure mode. **touchstone** is the foundation tool and validates Cortex stays out of Touchstone-managed write paths in the project that defines those paths. **vesper** is Swift, not Python, and validates language-agnosticism — stresses the audit against `Package.swift` (not `pyproject.toml`) as the dep-manifest source.

- [ ] **Install Cortex on conductor.** `cortex init` on the conductor source repo with its existing CLAUDE.md / AGENTS.md / README.md / `principles/` scanned-and-absorbed (uses v0.2.2's scan-and-absorb). Configure `.cortex/config.toml` `[audit-instructions]` section naming conductor's source-of-truth artifacts: `homebrew_tap = "autumngarage/conductor"`, sibling-repo references, PyPI presence (note: bare `conductor` on PyPI is unrelated). First-pass `cortex doctor --audit-instructions` run; capture findings as a baseline journal entry on conductor's `.cortex/`. Open install PR on `autumngarage/conductor`.
- [ ] **Install Cortex on touchstone.** Same shape on `~/repos/touchstone`. Configure `.cortex/config.toml` `[audit-instructions]` with `homebrew_tap = "autumngarage/touchstone"`, sibling-repo references including the conductor LLM-router cross-link in vanguard's CLAUDE.md (touchstone uses conductor for pre-push review). Verify Cortex stays out of Touchstone-managed write paths (`principles/`, `scripts/`, `.codex-review.toml`, `.pre-commit-config.yaml`). Open install PR on `autumngarage/touchstone`.
- [ ] **Install Cortex on vesper.** Same shape on `~/repos/vesper` — Swift project, different ecosystem. Audit must handle `Package.swift` (not `pyproject.toml`) as the dep-manifest source for T1.5. Configure `.cortex/config.toml` `[audit-instructions]` with `homebrew_tap` (if applicable — vesper currently distributes via direct download; capture this as a project-specific audit shape). Open install PR on `autumngarage/vesper`.
- [ ] **Sustained-work period across all three targets.** Every decision / incident / merge / release on conductor, touchstone, vesper goes through `cortex journal draft <type>`. `cortex next` consulted at every session start. `cortex doctor` run before every push. The Touchstone post-merge hook fires on every default-branch merge (target ≥ 5 auto-drafted `pr-merged` entries per repo to validate the hook under real load).
- [ ] **Surface bugs and friction in real-time.** Each significant friction (≥ 5 minutes of extra work, or any crash) gets a journal entry of `Type: incident` or `Type: decision` on the target's `.cortex/`, mirrored back to this repo's `docs/case-studies/` if it reveals a structural Cortex issue rather than a target-specific quirk. Bug fixes ship as v0.9.x point releases.
- [ ] **Exit-bar review.** End of sustained-work period: zero crashes on any target; non-trivial output from each command on each target's data; user assessment "I'd rather use this than hand-write" on each target. If any criterion fails on any target, scope a fix-and-retry pass before declaring the gate exit. Author a closure journal entry summarizing per-target findings and the cross-target pattern.
- [ ] v0.9.0 release.
