# 0007 — Canonical ownership of "where are we" and "what's next" lives in `.cortex/`, not at repo root

> Cortex's whole pitch is that `.cortex/` is the project's memory. The canonical answer to *"where are we right now?"* is `.cortex/state.md`. The canonical answer to *"what's next?"* is the active master plan in `.cortex/plans/`. Repo-root files (`ROADMAP.md`, `STATUS.md`, `PLAN.md`, `NEXT.md`, `TODO.md`) that restate this content are **anti-pattern** — they duplicate what `.cortex/` canonically owns and create the exact drift Cortex exists to prevent. README and other public-facing docs link to `.cortex/` files; they do not restate them.

**Status:** Accepted
**Date:** 2026-05-02
**Promoted-from:** `.cortex/journal/2026-05-02-canonical-ownership-doctrine-and-doctor-check.md` (the recursive-failure session — an agent attempting to resolve drift created `ROADMAP.md` at repo root, immediately producing the duplication failure mode the cleanup was supposed to fix)
**Load-priority:** always

## Context

On 2026-05-02, an agent session tasked with cleaning up documentation drift created a top-level `ROADMAP.md` as "the single page answer to when does Cortex launch." Within minutes, the same agent caught itself: the new file restated content already canonical in `.cortex/state.md` (current state) and `.cortex/plans/cortex-v1.md` (master plan). The act of cleaning up drift had created a fresh duplication surface — three places now claimed authority over the same facts.

This is exactly the failure mode Cortex exists to prevent at scale. Projects accumulate `ROADMAP.md`, `STATUS.md`, `PLAN.md`, `NEXT.md`, `TODO.md`, `docs/roadmap.md`, etc. over their lifetime. Each starts as a thoughtful answer to a real audience question. Each then drifts as the project's actual state changes faster than the file. The classic anti-pattern in legacy codebases is exactly this: five files claiming to be the source of truth for the same fact, none of them current, all of them load-bearing somewhere downstream.

Cortex's design is supposed to fix this by giving every project ONE canonical place for state (`.cortex/state.md`) and ONE canonical place for plans (`.cortex/plans/`). When a Cortex-using project then re-creates the legacy anti-pattern at repo root, Cortex's own design promise is broken.

The forcing event for this doctrine was Cortex doing it to itself. That makes the lesson doubly load-bearing: it caught the failure in dogfood, and the dogfood evidence is the doctrine's grounding.

## What we decided

**Single canonical owner per concern, enforced for `.cortex/`-using projects:**

1. **`.cortex/state.md` is the canonical answer to "where are we right now?"** Auto-regenerated by `cortex refresh-state`; hand-block (`<!-- cortex:hand --> ... <!-- cortex:end-hand -->`) preserves prose narrative through regeneration. Includes status-by-stage, active plans summary, recently shipped, open questions.

2. **The active master plan in `.cortex/plans/<slug>.md` is the canonical answer to "what's next?"** Each project has one master plan that owns the launch / next-milestone sequencing. Sub-plans are allowed and live in `.cortex/plans/` alongside, but reference the master for sequencing.

3. **Repo-root files that restate state or plans are anti-pattern.** Concretely: any of `ROADMAP.md`, `STATUS.md`, `PLAN.md`, `PLANS.md`, `NEXT.md`, `TODO.md`, `roadmap.md`, `status.md` at repo root duplicate `.cortex/`'s canonical content and create drift. If the audience for one of these files is "outsiders who won't read `.cortex/`," the right answer is to link from `README.md` to `.cortex/state.md` and the active plan, not to create a fourth surface.

4. **README and other public-facing docs link, never restate.** A short `## Status` paragraph with a one-liner of current shipped version + a link to `.cortex/state.md` and the active plan is correct. A multi-paragraph restatement of the launch sequence in `README.md` is wrong — it will go stale the moment the next release ships, and the README has no auto-regeneration affordance.

5. **Discovery affordance is a real concern.** First-time visitors arriving via GitHub do not naturally look in `.cortex/`. The fix is README links + a one-paragraph orientation, not duplication. Projects worried about discoverability should ensure their README is short enough that a reader gets to the link before tiring.

## Enforcement

`cortex doctor` warns (not errors) on:

- A repo-root file matching `^(ROADMAP|STATUS|PLAN|PLANS|NEXT|TODO|roadmap|status|plan|plans|next|todo)\.md$` when `.cortex/state.md` exists AND at least one `.cortex/plans/*.md` has `Status: active`.
- The warning text names the offending file(s), points to the canonical `.cortex/` files, and cites this doctrine.
- The warning is overridable per-project via `.cortex/config.toml` `[doctrine.0007]` `allowed_root_files = ["ROADMAP.md"]` for projects that have a real reason to keep one (e.g., a tool whose audience is overwhelmingly people who will never read `.cortex/`). The override is explicit, not silent.

This check ships in **v0.6.0** as part of the trimmed `cortex doctor` invariant expansions.

## Consequences

- **For Cortex itself.** `ROADMAP.md` deleted on the same session that created it (2026-05-02). README's `## Status` section slimmed to a one-paragraph link to `.cortex/state.md` + `.cortex/plans/cortex-v1.md`.
- **For projects adopting Cortex.** A common adoption mistake (creating a top-level `ROADMAP.md` "to keep things tidy") is now surfaced by `cortex doctor` instead of accumulating silently. The warning includes the citation to this doctrine so the project understands *why* it's flagged, not just *that* it's flagged.
- **For Cortex-using teams.** The single-source-of-truth discipline that `principles/documentation-ownership.md` already articulates as a *human* engineering principle now has a *machine* enforcement path for the specific case of state + plans. This is the load-bearing combination: the principle says what to do, the doctrine says where in `.cortex/` it must live, and the doctor check catches the regression.

## Bounds

This doctrine governs duplication of *project state and forward-looking plans*. It does not govern:

- Release notes / changelog — `CHANGELOG.md` at repo root is canonical; `.cortex/journal/` `Type: release` entries are the reasoning behind release decisions, not duplicates of the changelog.
- Architecture / code overview — `README.md`, `ARCHITECTURE.md`, `docs/` are appropriate canonical owners.
- Contributing guides — `CONTRIBUTING.md` is appropriate at repo root.
- Tool-specific READMEs in subdirectories — these own their subdirectory's surface, not the project's overall state.

The line: if the file's content answers *"where are we right now?"* or *"what's the launch sequence from here?"*, it duplicates `.cortex/`. If it answers *"how does this code work?"* or *"how do I contribute?"* or *"what shipped in version X?"*, it doesn't.
