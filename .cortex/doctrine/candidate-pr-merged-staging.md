# candidate — Stage T1.9 entries on the source PR; the post-merge hook becomes a verifier

> Cortex's first-party tooling should stage the T1.9 (`pr-merged`) journal entry on the source PR pre-merge, so the source PR's squash carries both the change and its journal record. The post-merge hook's role flips from primary writer to verifier. The Protocol's T1.9 contract — that every merged PR has a journal record — is unchanged; only the tooling defaults that produce the entry change.

## Context

Today's first-party tooling produces two PRs per substantive change:

1. The source PR — the actual feature/fix.
2. A follow-up auto-draft PR landing one journal file documenting the merge of (1).

The 2:1 meta-merge ratio is not abstract. In a four-day window (2026-05-04 → 2026-05-08) the cortex repo merged **six** auto-draft journal PRs (`docs/journal-pr-NNN`) alongside their source PRs. Half of `git log --merges` is meta. The autogenerated entries record *what* merged (extracted from the PR body the source PR already has) — not *why* the change was correct, what alternatives were rejected, or what surprised the author. The "why" detail is exactly what T1.9 was specified to capture (Protocol §2: "ties Plans, Journal entries written during the branch, and the final diff together at the moment ratification happened").

The post-merge writer surface has also been demonstrably fragile. PR #200 introduced the feature-branch + PR shape (closing #194); regressions and adjacent bugs followed in #203 (label-args unbound), #220 (SPEC_VERSION-missing exit 2), and #204 (commit-to-main + branch-protection). PR #226 just shipped the latest in the series. Each fix is small; the surface is small; the issues land disproportionately because the writer runs at the moment of least author attention (post-merge, branch already deleted, context already rolled forward).

A primitive that didn't exist when the post-merge writer was first wired now does: `cortex check-triggers` (issue #195, shipped via #199) returns structured NDJSON of which Tier 1 triggers fire on a given diff. That makes pre-merge gating mechanical — the same JSON the post-merge auditor consumes can drive a pre-merge staging step in `open-pr.sh`.

The Protocol's T1.9 wording — "Authored by whichever agent/human runs the merge command (or by a post-merge hook when present)" — is already neutral about *when* the entry is authored. Both shapes are spec-compliant. This doctrine is about tooling defaults, not Protocol contract.

### Three options weighed

1. **Status quo — post-merge hook writer.** Friction-free; runs without author action. But: 2:1 trunk noise; entries autogenerated from PR-body metadata, recording what not why; writer surface fragile (history of regressions); author's context cold by the time the entry exists.
2. **Pre-merge stage on the source PR.** `cortex journal stage --type pr-merged` writes the entry on the source branch; one squash subject ties feature + journal. Trunk noise halves. Author can edit the entry while the context is hot — this is the contribution that autogeneration cannot make. But: requires author opt-in unless integrated with `open-pr.sh`; changes the contract for `open-pr.sh` and `merge-pr.sh`; loses friction-free auto-write if treated as a manual step.
3. **Hybrid — stage if author runs `cortex journal stage`, else fall back to post-merge hook writer.** Combines the two write paths. But violates "One code path" (`engineering-principles.md`): the same trigger now has two writers, which drift apart silently. The "stage if I remember, otherwise auto" shape also reproduces the recall-burden problem that issue #195 was specifically designed to eliminate.

## Decision

**We will adopt option 2 — pre-merge staging on the source PR — with auto-invocation from `open-pr.sh` (and `merge-pr.sh` as a backstop). The post-merge hook's writer role retires; its replacement role is to *verify* a T1.9 entry was staged for any merge that fired the trigger.**

The boundary, specifically:

1. **`scripts/open-pr.sh` invokes `cortex check-triggers --staged` after pushing the branch and before opening the PR.** If a T1.9 trigger fires (always, for a PR-shaped merge against the default branch — the trigger is configurable per project per Protocol §2's "Trigger thresholds are project-configurable" clause, but defaults to every merge), the script invokes `cortex journal stage --type pr-merged --pr <num>` to write `.cortex/journal/<date>-pr-merged-<slug>.md` from the same PR-body / commit metadata the post-merge writer uses today, then `git add` + amend the open PR with a single commit so the squash subject becomes "feat(X): change AND its journal." The author can edit the file before merging.

2. **Authors retain the opt-out.** `open-pr.sh --no-stage-journal` skips staging; intended for trivial doc-only PRs where the change is its own record. The flag is loud (named, not silent) and `cortex doctor --audit` will still flag a missing T1.9 entry post-hoc — opt-out doesn't disable the audit.

3. **`merge-pr.sh` runs `cortex check-triggers --strict` on the about-to-merge branch as a backstop verifier.** If T1.9 fired and no staged entry is present, the merge is gated with a clear message naming the missing entry and the staging command. This catches the case where a PR was opened by a different tool path (e.g., GitHub UI) without staging.

4. **`scripts/cortex-pr-merged-hook.sh` (the post-merge hook) flips from writer to verifier.** Its responsibility becomes: after the squash lands on the default branch, confirm a T1.9 entry exists in the squashed diff that names the merged PR. If yes, exit 0 silently. If no, log a warning citing the strict-mode message and the command to backfill manually (a single deliberate action by the author, not an automated PR). The hook stops creating chore branches, stops committing on the default branch, and stops opening follow-up PRs altogether — eliminating the entire weak-point surface that produced #194 / #203 / #204 / #220.

5. **One transition release.** The next minor release ships both writer paths behind a `[journal.t1_9]` config switch in `.cortex/config.toml` defaulting to `mode = "stage"` (the new shape). Projects that prefer the old shape can set `mode = "post-merge-writer"` for one release. The release after that drops the post-merge writer; the verifier is the only post-merge role.

The Protocol stays unchanged. T1.9's contract — "every merged PR has a journal record" — is preserved; the *who/when* of authorship was already left to tooling by the parenthetical "(or by a post-merge hook when present)."

## Consequences

- **What becomes easier.** Trunk noise halves: one PR per substantive change, one squash carrying both the code and the journal. Authors edit the journal entry while the context is hot — the failure mode where autogenerated entries record *what* not *why* gets a real fix, not just a bigger template. The post-merge surface that produced four bug fixes in the last two weeks (#194, #203, #220, #204) collapses to a verifier with no commit-or-push side effects, eliminating the class.
- **What becomes harder.** `open-pr.sh` and `merge-pr.sh` gain a real responsibility (staging + verification); they're already non-trivial and this adds surface to maintain. Projects using the GitHub UI's merge button bypass `merge-pr.sh`'s strict verifier — the post-merge hook's warning is the only safety net there, and a warning is not a gate. Projects that consume the merged-PR commit log expecting the historical 2:1 shape need to migrate (the squash subject changes; the chore-branch journal PRs disappear).
- **What this forecloses.** The hybrid shape (option 3) is rejected as a long-term default; "stage if remembered, write post-merge if not" is incompatible with "One code path". The post-merge hook never returns to writer status — once it's a verifier, future T1 automation lives in `cortex check-triggers` callers (the §2 path), not in bespoke per-trigger writers. New Tier 1 triggers added to the Protocol follow this same shape: stage pre-merge from `open-pr.sh`, verify in `merge-pr.sh` strict mode.

## Bounds

This doctrine governs the *first-party tooling defaults* for T1.9 (`pr-merged`) authoring shape. It does not govern:

- **The Protocol's T1.9 contract.** The Protocol still permits a project to author T1.9 entries via a post-merge hook (its wording supports both); this doctrine recommends one tooling default, not the only valid path.
- **Other Tier 1 triggers.** T1.10 (release) has its own infrastructure (`release.md` template; auto-draft on tag-published event) and is not in scope here. T1.1–T1.8 use `cortex check-triggers` per #195's design; whether their writers stage pre-merge or post-action is a separate decision that this doctrine sets a precedent for but does not ratify.
- **Tier 2 triggers.** Judgment-based; not auditable. Out of scope.
- **Projects that intentionally generate trunk metadata noise** (e.g., a release-train workflow where every merged PR produces a release-note PR by design). The doctrine assumes the 2:1 ratio is unintentional cost; projects that rely on it can override via the `[journal.t1_9]` config switch or by keeping the verifier disabled.

The line: if the question is *"in what order does authorship happen relative to the squash, and where does it land?"*, this doctrine answers it. If the question is *"what is T1.9, and must it fire for every merge?"*, the Protocol answers it.
