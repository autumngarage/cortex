## What shipped

### Stale-checkbox detector — closes [cortex#100](https://github.com/autumngarage/cortex/issues/100) (PR #102)

`check_stale_plan_checkboxes` in `src/cortex/doctor_checks.py`. Always-on plain-`cortex doctor` check that warns when active-plan `- [ ]` checkboxes overlap with content from recent (≤14d, configurable per `.cortex/config.toml [doctor.stale-checkbox] window_days`) `Type: release` or `Type: pr-merged` journal entries' `## What shipped` sections.

Strong signals: shared `PR #N` references, shared `briefs/<name>.md` paths, shared doctrine numbers. Weak signal: shared distinctive 3-grams (≥2 to fire). Bypass: `<!-- cortex:no-stale-check -->` annotation on the checkbox line for legitimately-aspirational items (already used on the v0.7.0 S3 Sentinel-consumer line).

Catches the failure mode that opened the post-v0.8.0 session: `cortex next` returned already-shipped work as P0 because checkboxes weren't flipped during the v0.6.0/v0.7.0/v0.8.0 shipping marathon.

### Auto-draft substitution — closes [cortex#101](https://github.com/autumngarage/cortex/issues/101) (PR #104)

`cortex journal draft pr-merged --no-edit` now substitutes `{{ nnn }}`, `{{ short title }}`, `{{ full sha }}`, `{{ <type>/<slug> }}`, `{{ <date>-<slug> }}` from `--pr N` (or inferred from HEAD's `(#NNN)` merge subject) + `gh pr view --json` lookup. Touchstone post-merge hook auto-drafts now ship as useful T1.9 records, not template noise. `--pr N` flag added.

The three pre-fix canary entries (`2026-05-03-pr-merged-2039.md` / `-2057.md` / `-2119.md`) stay as raw-template artifacts per Journal-is-append-only (Protocol § 4.1) — they're authentic dogfood evidence of the bug. Canary entry #4 onward (post-v0.8.1 brew upgrade) will substitute.

### Append-only check false-positive — closes [cortex#103](https://github.com/autumngarage/cortex/issues/103) (PR #104)

Dropped `--follow` from `git log` invocations in `check_append_only_journal` and `check_immutable_doctrine`. Eliminates false positives where git's rename heuristic traced template-identical journal entries back to the template's modification commits in `.cortex/templates/`. Per Protocol § 4.1 + § 4.2, journal entries don't rename and doctrine entries don't rename — losing rename-trace is correct behavior, not a regression.

`cortex doctor` warning count on cortex's own repo: 9 → 7 (the 3 historical doctrine-mutation false positives + 1 canary-template false positive all gone; remaining 7 are the 4 pre-existing AGENTS.md/CLAUDE.md scope-qualifier warnings + map.md staleness + 2 canary-template false positives that stay until the next refresh). Wait — recounting against the actual `cortex doctor` output post-fix: 0 errors, 7 warnings, all pre-existing or expected.

### Schema-drift cleanup — closes [cortex#93](https://github.com/autumngarage/cortex/issues/93) + [cortex#94](https://github.com/autumngarage/cortex/issues/94)

Both fixed by PR #96 (which shipped in v0.8.0) but the issues stayed open through the v0.8.0 release because the PR's body cited them in prose without GitHub-magic `Closes:` trailers. Closed with citation as part of v0.8.1 housekeeping.

### Recursive-loop status

Three structural enforcement loops now live in cortex (each surfaced via dogfood, promoted to enforcement, ships to all Cortex-using projects):

1. **Stale external claims** → `--audit-instructions` + `Verified:` (v0.3.0/v0.5.0)
2. **Canonical-ownership drift** → Doctrine 0007 + canonical-ownership doctor check (v0.6.0)
3. **Stale plan checkboxes** → `check_stale_plan_checkboxes` (v0.8.1)

The third loop is the most directly recursive — it catches the very failure mode that triggered today's session ("`cortex next` ranking already-shipped work as P0").
