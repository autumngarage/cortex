[`plans/init-ux-fixes-from-touchstone`](../plans/init-ux-fixes-from-touchstone.md) Slice 1 (Sev-1 must-ship) shipped today as v0.2.4 (PRs #34 implementation + #35 release prep + Homebrew tap formula `9eb4eea`). ### What this proves about the design

The v0.2.3 → v0.2.4 fix validates a structural read: **Cortex's auto-absorb is only valuable when it adds content the agent doesn't already have.** When a file is already imported into CLAUDE.md / AGENTS.md via `@path`, re-absorbing it as Cortex Doctrine creates noise (stub-pointer entries), not signal. The Touchstone-detection primitive (already used for sibling-tool surfacing) is the load-bearing test for "does the agent already see this content via another path."

This generalizes beyond Touchstone: any project with first-class `@path` imports (Sentinel, custom in-repo conventions, future autumngarage tools) deserves the same treatment. v0.2.5 may extend the `_TOUCHSTONE_MANAGED_DOCTRINE_GLOBS` model to cover other detectable @-import sources, OR keep it narrowly Touchstone-scoped and let other patterns surface organically. Defer that decision to v0.2.5 design time.

## What this forecloses

**The "early dogfood" pattern is now part of Cortex's release process.** Yesterday's UX test (running v0.2.3 on touchstone before any v0.3.0+ features had shipped) caught two Sev-1 bugs that would have undermined the v0.9.0 official dogfood gate. The lesson: don't wait for the formal gate to test installs on the dogfood target. Each minor release between now and v0.9.0 should include a "fresh install on touchstone, see what breaks" pass — not as a ship-blocker, but as a cheap signal source. Future plans (`plans/cortex-v1` v0.3.0 → v0.6.0) should include a re-test-on-touchstone work item alongside their release-ritual slice.

**Cortex's Touchstone-detection primitive is now load-bearing for absorb decisions, not just for sibling surfacing.** Pre-v0.2.4, `_detect_sibling_signals()` was informational ("✓ .touchstone-config" in scan output). Post-v0.2.4, the same signal gates whether `principles/*` becomes Doctrine. That elevates the detection primitive's reliability requirement: a false-negative (Touchstone present but not detected) reintroduces the v0.2.3 regression; a false-positive (Touchstone absent but detected) suppresses legitimate Doctrine absorption. Audit the detection logic carefully if it ever changes (currently: any of `.touchstone-config`, `.touchstone-manifest`, `.touchstone-version` present at project root → Touchstone present). Add a regression test if a new signal is added or removed.

**Cortex-on-touchstone install is ready for commit when you decide.** The current state is that touchstone has an uncommitted Cortex install. Whether to commit it to touchstone's main and start writing touchstone development decisions to `~/Repos/touchstone/.cortex/journal/` is the next decision in the cortex-on-touchstone arc. v0.9.0's official dogfood gate assumes Cortex IS being used on touchstone; this install is the start of that. Suggested next step: commit the install (one PR on touchstone), then start using `cortex journal draft` (when v0.3.0 ships) for touchstone development decisions.
