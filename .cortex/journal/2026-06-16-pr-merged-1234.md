> fix(journal): run post-merge hooks through project-local cortex.

## What shipped

- Fixed the T1.9 dogfood failure where the merge hook saw a stale PATH `cortex`
  binary even though the source checkout had the compatible CLI available via
  `uv run cortex`.
- Kept the post-merge hook verifier-only in `journal.t1_9.mode = "stage"`:
  it now resolves a compatible `journal verify` command and verifies the staged
  entry before any legacy writer-mode branch creation can happen.
- Limited project-local `uv run cortex` fallback to stage/verify paths. The
  retired post-merge writer does not get a new fallback that would revive
  `docs/journal-pr-*` branch creation.
- Normalized journal writes to one final newline so source-PR staged entries do
  not get rewritten by pre-commit during the amend step.

## Why this shape

The right fix is command resolution at the stage/verify boundary, not another
post-merge writer path. That preserves the doctrine decision that the source PR
carries the T1.9 record and the hook only verifies it after the squash. The
alternative — letting `uv run cortex journal post-merge` create a fallback
entry after merge — would keep the old two-PR metadata loop alive and was
rejected.
