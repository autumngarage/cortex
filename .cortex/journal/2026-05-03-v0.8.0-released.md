## What shipped

### Slice S2 of `cortex retrieve` (PR #95, `61d40ee`)

Closes the v0.7.0 retrieval interface. Sub-plan: [`plans/cortex-retrieve.md`](../plans/cortex-retrieve.md).

- **`cortex retrieve --mode {bm25,semantic,hybrid}`** — BM25 (S1) + semantic (cosine over sqlite-vec embeddings) + hybrid (RRF k=60 fusion of BM25 + semantic). `hybrid` becomes the default once the embeddings table is populated, with a one-time stderr notice on first flip (no silent behavior change).
- **Stack:** `sqlite-vec` for vectors, `fastembed` for on-device CPU embeddings (`BAAI/bge-small-en-v1.5`, 384-dim, ~25 MB ONNX). Pinned in `pyproject.toml` under `[project.optional-dependencies] semantic` extra so the default install stays light; users opt in via `pip install 'cortex[semantic]'`.
- **Cache path:** forced to `~/.cache/cortex/models/` (XDG-respectful per council delta #4) over fastembed's permission-collision-prone default. Per-project fallback at `.cortex/.index/models/` if `~/.cache` is unwritable.
- **Schema migration:** indexes built at S1 (no embeddings table) keep working; embeddings backfill is incremental, gated on the same mtime+size invalidation as S1's chunks table (council delta #1 — no `git ls-tree`; uncommitted edits visible within one rebuild).
- **Bare-repo / no-silent-failures:** sqlite-vec import fail → BM25 fallback + visible stderr notice; fastembed import fail → same; model download fail → same; aarch64 Linux (no onnxruntime PyPI wheel) caught by all of the above. Doctor surfaces a runtime-availability warning when the user has `.cortex/.index/` (i.e. opted into retrieve) AND the deps are missing — fresh scaffolds without retrieve usage stay silent.
- **Stable JSON contract:** `cortex retrieve --json` emits `[{path, score, frontmatter, excerpt}]` regardless of mode. This is the file-contract Sentinel's Planner role consumes in S3.
- **14 new tests** in `tests/test_retrieve_semantic.py`: RRF math, doctor runtime gating, JSON schema stability, embedding cache path, probe cache, model pinning, CLI surface. 1 xfail documents an edge case (HOME chmod 000 crashes `cortex_model_cache_dir` instead of falling back to project-local cache) tracked as a v0.8.x follow-up.

Acceptance verified post-merge: `cortex retrieve "doctrine canonical ownership"` returns Doctrine 0007 hits at the top under both `--mode hybrid` and `--mode semantic`. JSON schema matches the Sentinel-consumer contract.

### Schema-validator drift fixes (PR #96, `e60e2b7`)

Both surfaced as honest in-doc notes by the config-schema-doc agent (PR #91); this PR closes the loop.

- **cortex#93 — `gh_release` schema-validated but never parsed.** Removed from `[audit-instructions]` schema in `check_config_toml_schema`. The audit uses `github_repos` (a list of repos) instead. Users who had `gh_release` set in config now see a clean `unknown key` warning rather than silent acceptance + zero behavior change.
- **cortex#94 — `[refresh-index]` consumed by CLI but absent from doctor schema.** Added the section to the validator with `candidate_patterns: optional-string-list` (mirroring `RefreshIndexConfig`). Typos like `candidate_pattern` (singular) now surface as warnings; type mismatches as errors. Two regression tests cover the typo + wrong-type paths.

`pyproject.toml` + `src/cortex/__init__.py` 0.7.0 → 0.8.0. Closes the v0.7.0 retrieve work (S1 at v0.7.0 + S2 at v0.8.0). Test count went 437 → 439 (2 new regression tests for #93/#94). Total: 439 passed, 1 xfailed.
