## 1. Frame: the non-negotiables this assessment serves

From `cortex_master_plan.md` as quoted in #459 and #457:

1. **The default is the official Slack SDK/Bolt.** Hermes must clear a real evidence bar to displace it.
2. **In every outcome, Hermes never owns memory, sessions, prompts, search, confidence, or the decision graph.** Any leakage of those into Hermes is a named disqualifier.

The Cortex-side contract being protected ([docs/hosted-ledger.md §Visibility boundary](../hosted-ledger.md), enforced by [`src/cortex/hosted/visibility.py`](../../src/cortex/hosted/visibility.py), shipped via #468 / PR #483):

- Hosted retrieval is **deny-by-default and fail-closed**: every retrieval requires an explicit non-empty `visible_source_ids` list; missing authorization raises, never broadens.
- The shared visibility CTE excludes sources/documents flagged `deleted`, `revoked`, **`slack_channel_excluded`**, or `repo_installation_revoked` (`_DENY_VISIBILITY_FLAGS` in `visibility.py`).
- Prompt packs, retrieval traces, and user-facing logs may contain only cited text that survived this boundary.
- #409 additionally requires that future connector visibility controls **reuse this deny-flag model rather than inventing a parallel ACL scheme**.

The question for this document: where does Hermes persist Slack message content, session state, and credentials; what residue would a *gateway-only* Hermes keep; and whether that residue can be made compatible with the boundary above.

## 2. What Hermes persists

### 2.1 Session store: `~/.hermes/state.db` (SQLite, WAL) plus on-disk files

Hermes persists **every conversation from every platform as a session with full message history**. The official developer doc is explicit: "Hermes Agent uses a SQLite database (`~/.hermes/state.db`) to persist session metadata, full message history, and model configuration across CLI and gateway sessions" ([developer-guide/session-storage](https://hermes-agent.nousresearch.com/docs/developer-guide/session-storage), [source in repo](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/website/docs/developer-guide/session-storage.md)).

Source-level confirmation at the pinned commit:

- The `messages` table stores `role`, **`content` (full plaintext)**, `tool_calls`, `tool_name`, `reasoning` fields, and `platform_message_id` per message ([hermes_state.py L477–496](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/hermes_state.py#L477-L496)). The `sessions` table stores `source` (platform), raw `user_id`, the system-prompt snapshot, token/cost accounting, and lineage ([L440–475](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/hermes_state.py#L440-L475)).
- Two **FTS5 full-text indexes** (`messages_fts`, `messages_fts_trigram`) index message content for cross-session search ([hermes_state.py L5–11, L76–81](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/hermes_state.py#L5-L11); [session-storage doc](https://hermes-agent.nousresearch.com/docs/developer-guide/session-storage)).
- The gateway's `SessionStore` writes transcripts to SQLite and **falls back to legacy JSONL transcript files when SQLite is unavailable** — persistence is the only path; there is no in-memory-only mode ([gateway/session.py L685–708](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/gateway/session.py#L685-L708)).
- A session index (`~/.hermes/sessions/sessions.json`) maps session keys — embedding platform, chat IDs, raw user IDs, channel names — to session IDs ([gateway/session.py L738–759](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/gateway/session.py#L738-L759)).
- On provider-side API failures (and preflight dumps), Hermes writes **`request_dump_*.json` files containing the full request body — i.e., the full message array including Slack-derived content — to the logs directory** ([agent/agent_runtime_helpers.py L1073–1142](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/agent/agent_runtime_helpers.py#L1073-L1142); call sites in [agent/conversation_loop.py L956, L2954, L3157](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/agent/conversation_loop.py#L956)).
- Outbound deliveries (including cron-initiated sends) are **mirrored back into the target session transcript** ([gateway/mirror.py L1–40](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/gateway/mirror.py#L1-L40)).

There is no ephemeral/no-persistence mode for gateway sessions. A repo-wide search at the pinned commit for `ephemeral` / `no_persist` / `incognito` / `disable_session` finds only an unrelated batch-runner option (`ephemeral_system_prompt`, [batch_runner.py L570](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/batch_runner.py#L570)). **Unverified at runtime** (source-search evidence only); verification path: run a Hermes gateway against a sandbox Slack workspace, send one message, and inspect `~/.hermes/state.db` and `~/.hermes/sessions/`.

### 2.2 Retention defaults: indefinite

This is the single most load-bearing fact in the assessment. Hermes's own default config says it plainly:

> "state.db accumulates every session, message, tool call, and FTS5 index entry **forever**. … Default false: session history is valuable for search recall, and silently deleting it could surprise users. Opt in explicitly."
> — [`hermes_cli/config.py` L2279–2293](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/hermes_cli/config.py#L2279-L2293)

- `sessions.auto_prune` defaults to **`False`** → retention is **indefinite by default** ([config.py L2290](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/hermes_cli/config.py#L2284-L2293)).
- When opted in, `retention_days` defaults to 90, enforced opportunistically at CLI/gateway startup at most once per 24h ([hermes_state.py L4429–4500](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/hermes_state.py#L4429-L4500); gateway call site [gateway/run.py L2106–2123](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/gateway/run.py#L2106-L2123)).
- Pruning touches **ended sessions only**; active sessions are never pruned ([hermes_state.py L3833–3886](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/hermes_state.py#L3833-L3886)).
- SQLite `DELETE` does not return disk pages; deleted message content **lingers in free pages until a VACUUM runs**, and VACUUM only runs when a prune actually deleted rows ([hermes_state.py L4475–4482 and config comment L2294–2300](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/hermes_cli/config.py#L2294-L2300)). For deletion-as-policy (#410's concern), "pruned" is not "gone" until VACUUM.

### 2.3 Deletion paths

Deletion is operator-initiated and best-effort, not contract-grade:

- `hermes sessions prune` (default 90 days, `--source` filter) and `hermes sessions delete <ID>` ([hermes_cli/main.py L11369–11377](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/hermes_cli/main.py#L11369-L11377); [user-guide/sessions](https://hermes-agent.nousresearch.com/docs/user-guide/sessions)).
- Programmatic `delete_session` / `delete_sessions` / `clear_messages` / `delete_empty_sessions`, which also remove on-disk transcript and `request_dump_*` files ([hermes_state.py L3627–3790](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/hermes_state.py#L3627-L3790)).
- There is **no per-channel or per-user deletion primitive** (prune filters by platform `source`, not by Slack channel), no tombstone/audit record of what was deleted, and no deletion SLA concept. #410's requirements (per-source retention windows, verified deletion, SLA) would have to be built *around* Hermes, not configured *into* it.

### 2.4 Credentials

- Slack credentials are two long-lived secrets: `SLACK_BOT_TOKEN` (`xoxb-`) and `SLACK_APP_TOKEN` (`xapp-`), stored **plaintext in `~/.hermes/.env`** ([gateway/platforms/slack.py L305–312, L743–749](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/gateway/platforms/slack.py#L743-L749); [Slack setup docs](https://hermes-agent.nousresearch.com/docs/user-guide/messaging/slack); install script `chmod 600` [setup-hermes.sh L333–340](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/setup-hermes.sh#L333-L340)).
- Multi-workspace mode merges **multiple workspaces' bot tokens as a comma-separated list in one env var feeding one process and one state.db** ([slack.py L760–840](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/gateway/platforms/slack.py#L760-L840); [docs](https://hermes-agent.nousresearch.com/docs/user-guide/messaging/slack)).
- Provider OAuth tokens live in `~/.hermes/auth.json` created atomically with `0o600` ([hermes_cli/auth.py L1087–1125](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/hermes_cli/auth.py#L1087-L1125)); MCP OAuth tokens in `~/.hermes/mcp-tokens/<server>.json`, also `0o600` ([MCP docs](https://hermes-agent.nousresearch.com/docs/user-guide/features/mcp)). A genuinely good disk-boundary sanitizer strips borrowed runtime secrets before they reach `auth.json`, failing closed for unknown sources ([agent/credential_persistence.py L1–30](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/agent/credential_persistence.py#L1-L30)).

Credential hygiene is competent for a personal tool: correct permissions, atomic writes, fail-closed persistence policy. What it is not: encrypted at rest, secret-manager backed, or multi-tenant.

### 2.5 Memory — Hermes owns one, and it is on by default

Hermes ships a curated persistent memory (`~/.hermes/memories/MEMORY.md` and `USER.md`) injected into the system prompt every session, written by the agent mid-conversation ([tools/memory_tool.py L1–57](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/tools/memory_tool.py#L1-L57)). `memory_enabled` defaults to **`True`** ([hermes_cli/config.py L1675](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/hermes_cli/config.py#L1675)) and `memory` is in the default toolsets ([toolsets.py L53](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/toolsets.py#L53)). A pluggable `MemoryManager` supports external memory-provider plugins ([agent/memory_manager.py L1–25](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/agent/memory_manager.py#L1-L25)). Under the non-negotiables, a default-on agent memory distilled from Slack conversations is a direct instance of "Hermes owns memory." It is disable-able (`memory_enabled: false`) — but the default leaks.

### 2.6 Media, attachments, and ambient caches

- Slack file attachments are downloaded with the bot token and written to `~/.hermes/cache/images/` / `cache/audio/` / document cache ([slack.py L3461–3501](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/gateway/platforms/slack.py#L3461-L3501); [gateway/platforms/base.py L560–730](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/gateway/platforms/base.py#L560-L640)). Image/document caches are swept hourly with a 24h TTL **only while the gateway is running** ([gateway/run.py L15449–15495](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/gateway/run.py#L15449-L15495)); a stopped gateway leaves the cache in place. Audio-cache sweeping is **unverified** (the ticker docstring claims it; the code section read covers image + document); verification path: read the rest of the ticker loop or run it.
- `~/.hermes/channel_directory.json` caches the reachable-channel map (Slack channel IDs and names) refreshed every 5 minutes ([gateway/channel_directory.py L1–20](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/gateway/channel_directory.py#L1-L20)).

### 2.7 Encryption posture and trust model

There is **no at-rest encryption** anywhere in the persistence inventory above — `state.db`, transcripts, `sessions.json`, `.env`, memories, and caches are plaintext guarded by POSIX file permissions. A repo-wide search for encryption primitives finds only platform-protocol crypto (WeCom callbacks, Matrix E2EE deps), not storage encryption. Hermes's own [SECURITY.md](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/SECURITY.md) is admirably explicit about this posture: Hermes is a **"single-tenant personal agent"**; "the only security boundary against an adversarial LLM is the operating system"; in-process screens are heuristics, not boundaries. Multi-user channels are explicitly named as a surface requiring whole-process sandboxing to stay within the supported posture.

This matters less as a criticism of Hermes (it is honest and coherent for its design center) than as a category statement: **Hermes's documented trust model is the opposite of hosted multi-tenant Cortex.** There is no `tenant_id` anywhere in the schema; isolation between two Slack workspaces sharing one gateway is a `source`/`team_id` column, not a boundary.

## 3. Gateway-only role: what flows through, and what residue stays

"Gateway-only" for Cortex means: Hermes terminates Slack Socket Mode, and Cortex owns everything cognitive. What would actually happen to the data:

**What flows through:**

1. **Inbound gating is decent at ingest.** In channels, Hermes by default processes a message only if the channel passes an optional `SLACK_ALLOWED_CHANNELS` whitelist and the message @mentions the bot, replies in a bot thread, or hits a `free_response_channels` override; messages failing the gate `return` before session handling and are not persisted ([slack.py L2332–2380](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/gateway/platforms/slack.py#L2332-L2380)). DMs always process.
2. **Thread-context backfill pulls in bystanders.** On first entry into a thread, Hermes calls `conversations.replies` and **prepends up to ~30 prior thread messages — written by users who never addressed the bot — to the triggering message** ([call site L2399–2412](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/gateway/platforms/slack.py#L2399-L2412); [`_fetch_thread_context` L3129–3230](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/gateway/platforms/slack.py#L3129-L3230)). That backfilled text becomes part of the user message and is persisted into the transcript and FTS index. There is **no configuration toggle** for this behavior at the pinned commit (source search; **unverified** at runtime — verification path: mention the bot mid-thread in a sandbox workspace and dump the resulting `messages` rows).
3. **Attachments** are fetched with the bot token onto local disk (24h-TTL cache).

**What residue stays (under defaults, per §2):**

| Residue | Location | Default lifetime |
|---|---|---|
| Full message transcripts (incl. thread backfill from non-consenting users) | `state.db` `messages` + 2 FTS indexes; legacy JSONL fallback | **Forever** |
| Session metadata: raw Slack user IDs, channel IDs/names, workspace ID, system-prompt snapshot | `state.db` `sessions`, `sessions.json` | **Forever** |
| Full LLM request bodies on provider 4xx | `request_dump_*.json` in logs dir | Until session deletion/prune |
| Outbound message mirrors | session transcripts | **Forever** |
| Agent memory distilled from conversations | `memories/MEMORY.md`, `USER.md` | Forever (until agent/user edits) |
| Slack attachments | `cache/images/` etc. | 24h while gateway runs; indefinite if stopped |
| Channel map | `channel_directory.json` | Rolling |
| Slack bot + app tokens | `.env` plaintext (0600) | Forever |

**The integration surface is built on the retention.** Hermes's designed way for an external brain (which is what Cortex would be) to consume gateway messages is `hermes mcp serve`, whose `messages_read` / `conversation_get` / `attachments_fetch` tools **read from `sessions.json` and `state.db` directly — explicitly without the gateway running** ([mcp_serve.py L1–27](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/mcp_serve.py#L1-L27), [`messages_read` L561–613](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/mcp_serve.py#L561-L613); [MCP docs](https://hermes-agent.nousresearch.com/docs/user-guide/features/mcp)). You cannot consume Hermes-gateway messages through Hermes's own surfaces without Hermes persisting them first. The persisted store *is* the API. That MCP surface carries **no authentication or per-source authorization** — any local MCP client (Claude Code, Cursor, any subprocess that can spawn `hermes mcp serve`) reads every conversation from every connected platform.

One mitigating note for fairness: Hermes has a `redact_pii` option that hashes user IDs before LLM context — but Slack is **excluded** from the PII-safe platform set because Slack mentions require raw `<@U…>` IDs, so the option never applies to Slack ([gateway/session.py L195–262](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/gateway/session.py#L195-L262); default `False` at [hermes_cli/config.py L1560](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/hermes_cli/config.py#L1560)).

## 4. Interaction with the Cortex visibility boundary

The shipped Cortex boundary ([visibility.py](../../src/cortex/hosted/visibility.py), [hosted-ledger.md §Visibility](../hosted-ledger.md)) assumes a single fact: **all retrievable Slack-derived text lives behind the deny-by-default CTE**, so flipping `slack_channel_excluded` on a source makes its content unreachable by `ask_ledger`, `decisions_for_diff`, and projection queries. A Hermes gateway breaks that assumption four ways:

1. **A second, unguarded corpus.** Everything Hermes relays also lands in `state.db` — plaintext, full-text-indexed, outside Postgres, outside the visibility CTE, outside RLS plans, readable via an unauthenticated local MCP surface and the Hermes web UI. `slack_channel_excluded` cannot reach it. Excluding a channel in Cortex would stop *ledger* retrieval while the same messages remain searchable in Hermes — the precise failure #459 names ("a gateway that retains Slack data outside that boundary would undercut it").
2. **A parallel ACL scheme.** Channel scoping in Hermes is `SLACK_ALLOWED_CHANNELS` / `free_response_channels` / `require_mention` in env/config.yaml ([slack.py L3606–3640](https://github.com/NousResearch/hermes-agent/blob/a72bb03757c0c925c686f9774eefc8dc5a77b329/gateway/platforms/slack.py#L3606-L3640)). Keeping that in sync with Cortex's `slack_channel_excluded` flags is exactly the "parallel ACL scheme" #409's acceptance criteria forbid. Any drift is silent over-collection.
3. **Retention semantics are inverted.** Cortex hosted is append-only events + immutable provenance snapshots with deletion handled as explicit policy (#410: tombstoning that survives event-hash chaining, deletion SLAs with verification). Hermes is mutable SQLite with optional 90-day pruning, ended-sessions-only, no per-channel deletion, no audit of deletion, and content lingering in free pages until VACUUM. A user/org deletion request under #410 could not be verifiably satisfied for the Hermes copy without bespoke tooling.
4. **Bystander ingestion exceeds consent.** Thread backfill persists messages from users who never invoked the bot. Cortex's Slack consent story (curate-on-request, channel exclusions resolved by the permission adapter *before* retrieval) has no Hermes-side equivalent; the backfill happens before any Cortex code runs.

## 5. Per-datum conclusion table

Verdicts use #459's scale: **acceptable** / **configurable-to-acceptable** / **disqualifying**, relative to the hosted Stage 3 Slack ledger console.

| Datum | Under a Hermes gateway | Under a Bolt gateway (official Slack SDK) | Trust-boundary verdict |
|---|---|---|---|
| **Slack message text** | Persisted plaintext + FTS-indexed in `~/.hermes/state.db` (and JSONL fallback), default forever; includes ~30-message thread backfill from non-consenting users; also in `request_dump_*.json` on provider errors; readable via unauthenticated local MCP | Held in process memory for the listener callback; persisted **nowhere** unless Cortex writes it — Bolt is a listener-dispatch framework with no message store ([bolt-python](https://github.com/slackapi/bolt-python), [Socket Mode docs](https://docs.slack.dev/tools/bolt-python/concepts/socket-mode/)); Cortex writes only into the ledger behind `visibility.py` | **Disqualifying** under Hermes defaults; config (`auto_prune`, 1-day retention, strict mention, allowed-channels) shrinks but cannot eliminate the store, and thread backfill has no toggle |
| **Permalinks / message references** (channel ID + ts, channel names) | `platform_message_id` in `messages`, channel/thread IDs in `sessions` + `sessions.json` + `channel_directory.json`, rolling and indefinite | In-memory event fields; persisted only as Cortex provenance (source spans, content-hash snapshots) | **Configurable-to-acceptable** — metadata, not content; still a stale-prone duplicate of what the ledger owns |
| **Slack user IDs** | Stored raw in `sessions.user_id`, transcripts, and `sessions.json`; `redact_pii` exists but is hard-excluded for Slack | In-memory; persisted only if Cortex records them (provenance-tagged, behind visibility) | **Disqualifying as shipped** for retention-with-PII; no Hermes configuration redacts Slack user IDs |
| **Tokens** (`xoxb-`, `xapp-`) | Plaintext `~/.hermes/.env`, chmod 600; multi-workspace tokens merged in one var/process | Supplied as env vars by the deployment platform (e.g. Railway secrets); Bolt does not write tokens to disk (file-backed installation stores exist only for the OAuth flow, unused with Socket Mode tokens — [FileInstallationStore](https://docs.slack.dev/tools/python-slack-sdk/reference/oauth/installation_store/file/index.html)) | **Configurable-to-acceptable** (operational hygiene either way), but Hermes adds a disk copy Cortex doesn't need |
| **Session transcripts / session state** | The core Hermes feature: unconditional, schema-central, the substrate of `/resume`, search, insights, and the MCP read surface; no off switch | No session concept; conversation state is whatever Cortex implements (and Cortex's answer is the ledger) | **Disqualifying by construction** — "Hermes never owns sessions" cannot be satisfied by configuration; only by forking out `SessionStore`, at which point the borrow has no value |

## 6. Ranked risks

1. **Structural session ownership (disqualifier).** The gateway cannot run without writing transcripts; the external-consumer API reads from the persisted store. Violates the non-negotiable directly. Evidence: §2.1, §3. *Severity: blocks "borrow" for hosted Stage 3.*
2. **Indefinite plaintext Slack content outside `visibility.py` (disqualifier under defaults).** `auto_prune: False`; FTS-indexed; no tenant concept; unauthenticated local MCP + web UI read paths. Evidence: §2.2, §2.7, §4.1. *Severity: undercuts the shipped deny-flag boundary the moment the first message flows.*
3. **Bystander thread backfill.** Persists up to ~30 messages per thread from users who never engaged the bot; no config toggle. Evidence: §3.2. *Severity: consent/privacy exposure Cortex cannot remediate from its side.*
4. **Parallel ACL drift.** Hermes channel gating must be manually mirrored from Cortex's `slack_channel_excluded`; #409 explicitly forbids a second ACL scheme. Evidence: §4.2. *Severity: silent over-collection on drift.*
5. **Deletion that can't satisfy #410.** Ended-only pruning, no per-channel/user deletion, no tombstones, free-page residue until VACUUM, no SLA. Evidence: §2.2–2.3. *Severity: makes org deletion requests unverifiable for the Hermes copy.*
6. **Default-on agent memory (disqualifier as shipped, config-fixable).** `MEMORY.md`/`USER.md` distilled from conversations into every system prompt. Evidence: §2.5.
7. **Residue side-channels.** `request_dump_*.json` full request bodies on provider errors; attachment caches surviving gateway shutdown; `channel_directory.json`; multi-workspace token/data co-mingling in one store. Evidence: §2.1, §2.4, §2.6.

## 7. Conditions for "borrow," stated per #459's acceptance bar

**No configuration of stock Hermes makes the gateway-only borrow safe for the hosted Stage 3 path.** The blocking findings are structural, not parametric: session persistence has no off switch (risk 1), thread backfill has no toggle (risk 3), Slack user-ID redaction is hard-excluded (table row 3), and the deny-flag boundary cannot reach `state.db` (risk 2). Configuration can shrink the residue window (`auto_prune: true` + `retention_days: 1`, `memory_enabled: false`, `strict_mention`, `SLACK_ALLOWED_CHANNELS`, disabling `mcp serve`/web UI, encrypted volume, per-tenant isolated `HERMES_HOME`) — but the result still has Hermes owning sessions, which the master plan rules out in *every* outcome. The only Hermes shape that satisfies the boundary is a fork that removes `SessionStore`/`SessionDB` from the gateway, which forfeits the maintenance benefit that justifies borrowing and inherits a 4,600-line state module's churn anyway.

If #457 nonetheless wants Hermes-derived value, the defensible scope is **reading Hermes's adapter code as prior art** for Socket Mode reconnect handling, thread-session keying, rate-limit backoff, and multi-workspace client routing (`gateway/platforms/slack.py` is genuinely battle-tested) while implementing on Bolt — code-reading, not code-borrowing.

## 9. Input to #457 (summary paragraph)

For the build-vs-borrow decision: **borrowing Hermes as the Slack gateway is disqualified on retention grounds; the default — official Slack Bolt — stands.** Hermes (NousResearch/hermes-agent v0.16.0, pinned `a72bb037`) is a well-engineered *single-tenant personal agent* whose gateway unconditionally persists every relayed conversation — full plaintext message content, FTS-indexed, raw Slack user IDs, plus up to ~30 backfilled thread messages from users who never addressed the bot — into `~/.hermes/state.db` with retention defaulting to *forever* (`auto_prune: False`), no at-rest encryption, no tenant concept, no per-channel deletion, and an unauthenticated local MCP read surface that is itself built on the retained store. That store sits entirely outside the fail-closed `visibility.py` boundary (PR #483), so a `slack_channel_excluded` flag in Cortex would not stop the same messages remaining searchable in Hermes; channel scoping would require a parallel ACL that #409 forbids, and #410's deletion-with-verification could not be satisfied for the Hermes copy. These are structural properties (session persistence and thread backfill have no off switches), so no configuration satisfies the "Hermes never owns sessions or memory" non-negotiable — only a fork that removes the session store would, which forfeits the point of borrowing. Recommended boundary for #457: Bolt (Python) on Socket Mode as the gateway, Cortex ledger as the only persistence behind `visibility.py`; mine `gateway/platforms/slack.py` for reconnect/threading/rate-limit prior art as reading material only.
